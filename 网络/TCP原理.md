# T一、 宏观视角：生产者-消费者模型

## 核心架构：内核缓冲区的双向队列

在生产者-消费者模型中，核心组件是一个**阻塞队列**，在TCP中，这个队列被拆分成两个部分，分别位于发送端和接收端的OS内核中

* **生产者：发送端的应用程序**

  * 当调用`write`或者`send`时，并不是把数据发送到了网络上，而是**将数据总用户空间拷贝到了内核中的发送缓冲区**
  * 一旦拷贝完成，`write`系统调用就会返回，生产者任务暂时结束，至于数据什么时候发、发多少，完全由TCP协议栈决定

* **消费者：接收端的应用程序**

  * 数据从网卡驱动到达内核后，**拷贝到接收缓冲区**
  * 接收端的`read`或`recv`调用，本质是**从内核的接收缓冲区里拷贝数据到用户空间**

* **全双工通道**

  TCP是全双工的，意味着每一段既是生产者，也是消费者。每个TCP连接在内核中都有**两个缓冲区**，一个发送，一个接收

## 面向字节流的解耦特性

生产者-消费者模型的巨大优势是**解耦**：生产者的生产速率和消费者的消费速率不需要完全一致：

* **生产和消费的不对等：**

  * 生产者：可以一次性`write`100字节的数据
  * 消费者：可以分100次，每次`read`1个字节，也可以一次性`read`100个字节

* **底层原理：**

  因为由内核缓冲区的存在，消费者不需要知道生产者是怎么发的，只管从缓冲区里拿数据就好，只要缓冲区里有数据，就能拿，这种流式处理就消除了数据包边界的限制

**这样设计会导致什么问题？**

> 面向字节流，流式处理就消除了数据包边界的限制，对于TCP来讲，它的工作是**安全可靠地把应用层发出去的字节，发到内核缓冲区哪里**，TCP视角里没有包的概念，只有像水一样的字节

**如何解决粘包问题呢？**

> TCP消除了边界，**应用层必须自定义协议来恢复边界**
>
> * **固定长度**
>
>   约定好每个逻辑消息都是固定的长度
>
>   * 协议：每个消息必须是Y个字节
>   * 发送：`"XXXXXX"`->发送`"XXXXXX    "`（补齐到Y字节）
>   * 接收：消费者每次`read`必须攒够Y字节才处理
>   * 缺点：不灵活
>
> * **特殊分隔符**
>
>   在每个消息末尾加上特定分隔符，分隔符作为边界
>
>   * 协议：消息以`\n`结尾
>   * 发送：发送`"hello\n"`，再发送`"socket\n"`
>   * 接收：在缓冲区中寻找`\n`，找到`\n`说明截取到一个整整消息
>   * 缺点：消息内容本身你不能包含分隔符
>
> * **TLV格式**
>
>   在真正的消息体前面，加上一个固定长度的报头，报头里记录了后面消息体的长度
>
>   * 协议：[4字节长度] + [N字节消息体]
>   * 发送：发送`0005Hello`（0005是报头，代表后面有五字节）
>   * 接收：攒够4字节，解析出长度N，根据N，再攒够N字节，这N个字节就是纯净的消息体

## 传送带角色

在这个模型中，连接两个内核缓冲区的，是不可靠的**网络层**，就像一条不稳定的传送带：

* 可能会丢包
* 可能会乱序到达
* 可能会网络拥塞

那么TCP就需要一套机制来保证**数据不丢失（可靠性）、不乱序（有序性）且不爆仓（流量控制）**



# 二、 微观视角：解剖TCP报头结构

![image-20260302165627582](https://gitee.com/binary-whispers/pic/raw/master///20260302165629528.png)

## 报文的身份证

在应用层来看，数据是一段完整的消息（如"hello"），在传输层，这段数据会被打上一个**TCP报头**，封装成一个**TCP报文段**

* 标准尺寸：TCP报头有一个**固定长度：20字节**
* 扩展能力：TCP可以通过**选项字段**扩展，报头最长可以达到**60字节**
* 结构逻辑：按照32位对齐，像砌砖一样一层一层

## 核心字段的详细作用

* **第一层：身份识别（寻址）**
  * 16位源端口号&16位目的端口号
    * 作用：解决分用问题
    * 解析：IP地址解决了**给哪台机器**的问题，端口号解决**发给这个机器上的哪个进程**的问题
* **第二层：可靠性的基石（顺序与确认）**
  * 32位序列号
    * 作用：解决**乱序**和**去重**问题
    * 解析：TCP给发送的**每一个字节**都编了号，对方收到一堆乱序的包，可以根据序列号排队，还原出正确的数据
  * 32位确认应答号
    * 作用：解决丢包问题
    * 解析：表达含义是**我收到了该序号之前的所有数据，下次请从这里开始发**，比如给发送端发送`1001`，表示`1000`及以前的数据都收到了，请从`1001`开始发
* **第三层：连接控制和管理**
  * 4位首部长度
    * 为什么存在？-> 因为有**选项字段**存在，报头长度不固定
    * 计算方法：这个字段是**4字节**
      * 如果该值为5，表示报头长度为$5 \times 4 = 20$字节（标准长度）
      * 最大值为15，表示报头最长为$15 \times 4 = 60$字节
    * 所以：$报头长度 = 4位首都长度 \times 4 + 额外选项$ -> [20,60] 
  * 6位标志位
    * 这是TCP的控制面板，决定了当前报文的性质，服务器收到不同类型的报文，做出不同的动作
      * `SYN`：建立连接的请求
      * `FIN`：断开连接的请求
      * `ACK`：表明自己是一个确认**报文**，告诉对端“你要关心确认序号！”因为有捎带应答，所以大部分TCP报文的ACK都是1
      * `RST`：连接异常，强制断开连接
      * `PSH`：催促接收端尽快把数据读走，别在缓冲区墨迹
      * `URG`：说明包里有紧急数据，需要插队
  * 16位窗口大小
    * 作用：实现流量控制	注意：**流量控制!=减少发送**
    * 解析：标识**对方接收缓冲区的剩余空间大小**，接收方通过这个字段告诉发送方：我还能吃得下多少，你悠着点来，别把我撑着或者饿着了
* **第四层：数据完整性**
  * 16位校验和
    * 作用：保证数据没坏，发送端填充，接收端校验，如果校验不通过，直接丢弃，当作没收到
  * 16位紧急指针
    * 作用：标识哪部分数据是紧急数据

* 主机B要给主机A发的消息做出应答
* 主机A要给主机B的应答继续应答！
* 这样不就死循环下去了？

所以：**长距离通信，是没有100%可靠性的，总会有最新的一条消息没有得到应答，老消息有应答就是100%可靠的**

## 序列号：解决乱序和去重问题

如果网络卡顿，对方重发了数据，结果之前没收到的数据包又到了，接收方岂不是收到了两份同样的数据包？

* 痛点：网络层会导致数据包**乱序**到达，或者因为重传产生**重复**数据
* 原理
  * 排序：接收缓冲区就是一个`char`类型的数组，数据包到达之后，内核会根据TCP报头的**32位序列号**，把它放到对应索引的位置上，即使后发的包先到了，也只能等着，直到前面的空缺被填上
  * 去重：如果内核发现收到的数据包序号在缓冲区里已经有了，就会直接**丢弃**这个重复的报文段，通常会回复一个ACK，告诉对方“不用发了，我已经有了”

## 确认应答(ACK)：不仅仅是收到

**我发了一堆数据，怎么知道对方收到了哪部分？丢了哪部分？**

原理：**累计确认**
* TCP不会为每一个字节单独发送一个ACK，串行效率太低了
* **确认序号含义：我已经成功收到了该序号之前的所有字节，下一次发送请从该序号开始发送**
* 举例：
  * 发送方发了 1-1000, 1001-2000, 2001-3000 三个包
  * 接收方收到了 1-1000 和 2001-3000，**中间的 1001-2000 丢了**
  * 此时接收方**不能**回复 ACK 3001，只能回复 **ACK 1001**。
  * **含义：** 1001 之前我都收到了，但我想要 1001，请你重传

## 超时重传

![image-20260302175308665](https://gitee.com/binary-whispers/pic/raw/master///20260302175311231.png)

因为网络拥堵问题，A给B发送失败或B给A的应答A没收到

这个超时的时间如何确定？

> * 绝对不能写死
>   * 写长了业务层阻塞时间过长
>   * 写短了稍微有点网络波动就会引起一大批主机重发
> * 正常情况：动态计算内核会动态采样，每次数据包一来一回，内核会测量真实的耗时，动态更新下一次的超时时间，做到**网快就短，网慢就长**
> * 极端情况：指数退避，真的发生超时，说明网络极度拥塞。TCP会把下次超时时间**翻倍**，如果连续重试15次依然失败，底层Socket就会抛出`ETIMEDOUT`错误，通知业务层连接断开

## 快速重传

超时重传是**计时器驱动**机制，计时器时间如果设置过长，整个发送窗口会长时间卡死，拥塞窗口会断崖式跌到1

所以，快速重传是**数据驱动**机制，在超时之前提前感知丢包并补救

**触发机制：**

> 依赖于**累计确认**机制：
>
> 假设我发送了序号1~5的5个包，如果包2在路上丢了，但是345都到了，接收方只能向我发送`ACK=2`，我**连续收到了3个重复的`ACK=2`**（加上原本正常的那个，一共是4个相同的ACK）时，内核就会判定：包2确实丢了！此时不等计时器，立刻把包2重发

**为什么一定是3次冗余的ACK？**

> 这是为了区分丢包和乱序：
>
> IP报文在底层路由器转发时，走的路径可能不同，经常发生序号靠后的包先到，序号靠前的包后到的乱序情况，如果乱序了，接收端也会恢复重复的ACK
>
> 但是经过统计：如果是乱序，通常产生一两个重复的ACK后，那个走的慢的包也就到了，**但如果连续产生3个冗余的ACK，说明后续的包都到了好几个了，那个没到的包大概率是丢了**

## 连接管理（超级重要）

首先建立一个认知：**物理层面上，根本不存在所谓的连接**，网络是一根根网线和路由先，它们只负责转发电信号。而所谓的“TCP”连接，本质上是**通信双方的操作系统内核里，各自开辟了一块内存，维护了一个`struct tcp_sock`的状态机。建立连接，就是双方同步初始化这个状态机的过程；断开连接，就是双方销毁这块内存的过程**

### 三次握手

#### 详细过程

* 第一次握手（客户端发）

  * 报文：客户端发送一个`SYN`标志位为1的报文，说：**我要建立连接，我的初始序列号是x**
  * 内核：客户端调用`connect()`，触发底层发包，此时客户端TCP状态机处于`SYN_SENT`状态（发出就转换状态）

  ![image-20260302195955105](https://gitee.com/binary-whispers/pic/raw/master///20260302195957637.png)

* 第二次握手（服务器响应）

  * 报文：服务端收到后，恢复一个标志位`SYN`和`ACK`为1的报文，说：**收到你的x了(ACK=x+1)，我也要和你建立连接，我的初始序列号是y(SYN=y)**
  * 内核：服务端收到`SYN`，TCP状态机处于`SYN_RCVD`状态

  ![image-20260302200543189](https://gitee.com/binary-whispers/pic/raw/master///20260302200545367.png)

* 第三次握手（客户端拍板）

  * 报文：客户端收到了服务端的`SYN+ACK`，最后回复一个`ACK`报文，说：**收到你的y了(ACK=y+1）信道正式建成！**
  * 内核：客户端状态机进入`ESTABLISHED`（发出就转换状态），同时，服务器进入`ESTABLISHED`状态（发出就转换状态），此时服务器阻塞在`accept()`的代码唤醒，返回一个全新的Socket文件描述符

  ![image-20260302201045373](https://gitee.com/binary-whispers/pic/raw/master///20260302201048873.png)

注意：

* **三次握手的过程是操作系统完成的，`connect`和`accept`不参与！**
* **三次握手本质就是赌，赌最后一个ACK对方收到了**

再来一张图总结：

![image-20260302202747655](https://gitee.com/binary-whispers/pic/raw/master///20260302202750277.png)

#### 为什么要三次握手？

* **防范历史失效连接的内存泄漏：**

  假设网络极度堵塞，客户端发了`SYN1`卡在半路，超时之后又发了`SYN2`，如果只有两次握手，服务器收到迟到的`SYN1`直接分配内存建立连接，但是客户端不认这个旧连接了，导致服务器的Socket资源浪费

  而三次握手的精髓就是**客户端最后一次校验**，服务器回复`SYN1+ACK`给客户端，客户端发现上下文不对，我应该收到`SYN2+ACK`啊，立刻发送一个`RST`标志位为1的报文，强制把服务器建立好的半成品连接销毁

* **保证了全双工信道的通畅性：**

  客户端发`SYN`，服务端收到发来`SYN+ACK`，这里验证了客户端O的能力正常，服务器的IO能力正常，客户端收到`SYN+ACK`，发一次`ACK`，验证了客户端的I能力正常，你来我往，同时也验证了网络的通畅性！

### 四次挥手

#### 详细过程

* 第一次挥手（客户端发起关闭）
  * 报文：客户端发送`FIN`报文，说：**我的发送缓冲区空了，不再发数据了**
  * 内核：调用`close`，进入`FIN_WAIT_1`状态
* 第二次挥手（服务器确认，进入半关闭）
  * 报文：服务器回复`ACK`，说：**知道了，你不会再发了**
  * 内核：服务器收到`FIN`后，进入`CLOSE_WAIT`状态，客户端收到`ACK`，进入`FIN_WAIT_2`状态。此时是**半关闭状态：**客户端不能发，但是还能收，服务器应用层如果调用`read`回读到0，但是它底层发送缓冲区里的历史数据还会继续往外发
* 第三次挥手（服务器彻底发完）
  * 报文：服务器把剩下的数据发完，也向客户端发送`FIN`报文，说：**我的数据也发完了，我也关了**
  * 内核：服务器应用层调用`close`，进入`LAST_ACK`状态
* 第四次挥手（客户端最后确认）
  * 报文：客户端收到`FIN`，回复`ACK`，说：**收到，就此别过**
  * 内核：服务器收到`ACK`后，立刻进入`CLOSED`，销毁，**而此时客户端绝对不能立刻销毁**，必须进入`TIME_WAIT`状态，在内核中等待2MSL(60s)后，等游离的报文消散，再进入`CLOSED`

完整图解：

![image-20260302213832499](https://gitee.com/binary-whispers/pic/raw/master///20260302213909757.png)

双方`FIN`和`ACK`有来有回，连接断开成功

注意：**主动关闭连接的一段，才有`TIME_WAIT`状态**

#### 为什么是四次挥手？

服务器收到客户端的`FIN`时，**仅仅代表了客户端不发了，但服务器可能还有没发完的数据**，所以服务器必须回一个`ACK`稳住客户端（防止客户端超时重传`FIN`），等服务器自己的应用层把数据全部发完之后，才能发出属于自己的`FIN`。`ACK`和`FIN`中间隔着业务处理的时间，通常无法合并，必须四步走，确保双方的读写缓冲区都关闭了

### 理解`TIME_WAIT`状态

**`TIME_WAIT`到底是什么？**

> `TIME_WAIT`并不是异常状态，而是TCP四次挥手过程中，**主动关闭的一方**在发完最后一个`ACK`报文后，必须强制停留的一个等待状态。Linux下，通常是**2MSL(Maximum Segment Lifetime)**，报文最大生存时间的2倍（1min），然后才会彻底销毁内核中的Socket

**为什么发完最后一个确认包了，不立刻释放内存，还要等1min？**

> * **保证最后一个`ACK`到达（兜底对端的重传）：**
>   * 假设主动关闭方完成四次挥手就立刻销毁，此时网络抖动，这个`ACK`就丢了
>   * 被动关闭方等不到`ACK`了，只能超时重传第三次挥手的`FIN`，但是主动关闭方的Socket已经销毁，内核只能回复`RST`重置报文，被动关闭方收到`RST`就会抛异常
>   * 核心作用：**处于`TIME_WAIT`状态，就是为了兜底，在1min内，如果对方没收到`ACK`而重发了`FIN`，我还能再补发`ACK`**
> * **防止迷走报文污染新连接：**
>   * 网络中会有很多拥塞在半路迷路的老数据包，假设主动关闭方立刻销毁，并且瞬间用**完全相同的端口号**重新创立了一个新连接（相同的五元组：源IP、源端口、目的IP、目的端口、协议）
>   * 此时，上一个迷路的老数据包突然到了，新老连接的五元组完全一直，内核会把“前朝的剑”当成“本朝的数据”塞给应用层，导致数据错落
>   * 核心作用：**1MSL是报文在网络中最长存活时间，一来一回就是2MSL，等待时间足以保证上一个连接在这个网络中产生的所有迷走报文全部消亡，不会污染紧接着启动的新连接**

**怎么避免这种情况，我就想断开后立刻复用源端口**

```cpp
int sockfd = socket(AF_INET, SOCK_STREAM, 0);

int opt = 1;
// 设置 SO_REUSEADDR，允许复用处于 TIME_WAIT 状态的本地地址和端口
setsockopt(sockfd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

// 然后再绑定，绝对不会再报 Address already in use
bind(sockfd, (struct sockaddr *)&addr, sizeof(addr));
```

## 滑动窗口（超级重要）

### 发展推演

TCP每发送一个数，都要进行一次应答，上一个数据包收到了应答，再发送下一个，这样串行效率太低！

那么我们就可以一次性发一批

但是这种思路只考虑到了自己，没考虑到对端，对端的接收缓冲区满了，我发的数据包就会被丢弃，引起我疯狂重传，直到把网络干崩

**滑动窗口就解决了发送速率和接收方处理能力不匹配的问题**

### 窗口设计

在物理层面，根本没有窗口这个实体，本质是**双方OS内核中维护的几个指针，来圈定内核缓冲区里的一段连续内存空间**

:black_nib: 这种指针分割的方法和虚拟地址空间设计思路大致相同

#### 接收端窗口

逻辑上是一个**环形队列**，这样就可以用有限的内存空间接收无限的数据包

接收端窗口大小 = **缓冲区总大小 - 已经收到但还没被应用层取走的数据大小**

接收方每次回复`ACK`时，都会把这个计算出剩余空间大小，填入**16位窗口大小**字段，广播给发送方，意思是：**老哥，我还能吃得下X字节，你接下来的发送量千万别超过这个数！**

![image-20260303104122834](https://gitee.com/binary-whispers/pic/raw/master///20260303104124508.png)

#### 发送端窗口

* **已发送且已确认：**这部分数据已经安全收到，内存可以随时回收
* **已发送但未确认：** 这部分数据正在网络中游，或者对方收到了还没回ACK
* **允许发送但尚未发送：** 在接收方的承受范围内，随时可以调用底层网卡发出去
* **未发送但超过对方处理范围：** 超出接收方承受范围

![image-20260303114204323](https://gitee.com/binary-whispers/pic/raw/master///20260303114206134.png)

### 窗口是怎么滑动起来的？

由**接收方发来的 ACK 确认报文驱动**

当发送端收到一个ACK报文，假如说是`ACK = 100 win = 200`，内核会做两件事：

* **确认历史数据：** 将底层已发送未确认的最老字节指针向右移动到100，这意味着前99字节已经被对端应用层使用，这部分内存可以被回收或覆盖
* **吐出新的发送额度：** 根据对端通告的`win = 200`，重新计算并向右推移窗口的右边界。左边界和右边界整体向右滑动，发送缓冲区会吐出一段全新的可用窗口，这时候，底层网卡才有权限把后面的数据发出去

### 如果丢包了怎么办，害怕丢包吗？

为了配合重传机制解决丢包问题，滑动窗口里专门维护了一个已发送但未确认的区域，这块内存充当了物理备份库

把数据发给网卡之后，发送端内核绝不会立刻销毁这块内存。因为底层网络是不可靠的，在没有收到对方的 ACK 回执之前，这块内存必须强制保留在已发未确认区域，作为应对网络丢包的物理备份库

如果真的丢包了：

* **发送的数据丢了：** 接收端会根据**累计确认**，假设发来了101~110连续的10个字节，中间就105一个包没收到，其它都收到了，这时候接收端会向发送端发送`ACK = 105`，说：”105之前的4个包我全收到了，接下来从105开始发！“，发送端窗口的左边界会因为等不到新的 ACK 而**彻底停滞**。随着业务层继续发包，可用窗口会被迅速榨干。当发送端连续收到 3 个 `ACK = 105` 时，就会触发**快速重传**，内核立刻从刚才的备份区域把 105 重新发出去。一旦收到新的确认，停滞的窗口就会瞬间向右滑动，吐出新的发送额度
* **对方回的ACK丢了：** ACK的含义是**我已经成功收到了该序号之前所有内容，下次请从这个序号开始发**，即便是中间某个 ACK（比如 ACK 105）在路上丢了，只要后续的 ACK（比如 ACK 110）成功抵达，内核就会根据累计确认原则，直接无视掉那个丢失的 ACK，顺滑地将发送窗口向右推进

## 流量控制和拥塞控制

先明确一个大的方向，**TCP如何控制发送速度？**

> TCP控制发送速度有两套独立但又相互制约的机制：
>
> * **流量控制关注接收方的消化能力**，是端到端的问题
> * **拥塞控制关注的是整个网络链路的能力**，是全局问题
>
> TCP最终的发送速度，取决于二者的最小值

### 流量控制

**让发送方根据接收方的实际接收能力控制发送的数据量，这就是所谓的流量控制**

这里需要注意：**控制 != 减少**，对方能力大，就增大发送量，对方能力小，就减少发送量

而滑动窗口的本质就是实现流量控制的机制

### 拥塞控制

**网络是一个黑盒，发送端根本不知道中间的带宽有多大。** 它唯一的试探手段就是：**只要没丢包，就说明网还通（踩油门）；一旦丢包，就说明网堵了（踩刹车）**

内核维护了一个`cwnd`拥塞窗口变量，它有四个阶段：

* **慢启动：探底的指数飙升**

  * 动作：初始`cwnd = 1`，**每次收到一个合法ACK，`cwnd += 1`**
  * 推演：发1个包，收1个ACK，`cwnd`变为2->接着发2个包，收到2个ACK，`cwnd`变为4... 这是一个**指数级增长$2^n$**，”慢“的原因是起点低，但是加速快，旨在快速摸清楚网络状况

* **拥塞避免：线性逼近**

  * 触发：指数飙升太危险，内核设置了一个阈值（假设为`Y`），当`cwnd >= Y`时，切换为线性增长
  * 动作：**每经过一个完整的往返时间，`cwnd`才加1**，这时候就缓慢逼近界限

* **踩刹车分化：急刹VS点刹**

  * **触发超时重传->急刹**

    * 底层判断：包和ACK都没了，网络大概率严重拥塞
    * 动作：
      * 阈值暴跌：`Y = cwnd / 2`
      * 窗口回到起点：`cwnd = 1`
      * 状态切换：重新回到慢启动阶段

  * **触发快速重传->点刹**

    * 底层判断：收到了后续的ACK（比如连续收到了3个`ACK=105`），说明网络只是抖动了一下

    * 动作：

      * 阈值减半：`Y = cwnd / 2`

      * 窗口微调：`cwnd = Y + 3`

        * 为啥加3？

          **收到了3个冗余的ACK，说明有3个包虽然乱序，但确实已经离开网络到达接收端了！网络腾出了3个包的位置，那么我的窗口就能加2了**

      * 状态切换：只要继续收到冗余 ACK，`cwnd` 就线性加 1。直到收到全新的正常 ACK，说明拥堵扛过去了，`cwnd` 恢复到 `Y`，继续执行线性的拥塞避免

# 四、TCP和UDP

## 差异对比

**连接状态与维护机制**

> TCP收发数据之前必须通过三次握手在双方内核建立`tcp_sock`状态机，那么就必须调用`listen`和`accept`
>
> UDP是轻量、无状态的，不需要建立连接，只需要`bind`一个端口，随时可以向任何IP扔数据

**可靠性和开销**

> TCP为了保证绝对可靠，报头有20字节，并且实现了复杂的序列号排队、超时重传和滑动窗口机制
>
> UDP只有精简的8字节，它不关心丢包和乱序，把可靠性的烂摊子全部交给了应用层

**数据的边界**

> TCP面向字节流，底层像水流一样把数据匀过去，导致了业务层的粘包问题，必须手写协议来拆包
>
> UDP是面向报文的，严格保留了消息边界，调用一次`sendto`就对应一次`recvfrom，绝对不会粘包`

## UDP不可靠，是真的不可靠吗？

> 完全不可靠是不严谨的
>
> UDP只是不保证数据能到达和能按需到达，但是一定能保证数据**没有损坏**
>
> UDP报头里，有**16为校验和字段，当UDP数据包在公网链路中因为物理干扰发生比特位反转时，接收端的内核校验计算就会失败**，**一旦校验失败，直接丢包**，UDP是尽力而为，而不是瞎搞传错数据



## 一台服务器，TCP服务端和UDP服务端，能同时监听同一个端口吗？

> 可以！协议栈在路由一个数据包时，根据**【协议、源IP、源端口、目的IP、目的端口】这个五元组来确定一个Socket的**
>
> TCP和UDP的协议类型字段在IP报头里是不同的
>
> 因此，内核完全可以清晰地把发往相同端口的 TCP 包交给 TCP 监听队列，把 UDP 包交给 UDP 的接收队列，它们在操作系统内部对应的是两棵完全不同的路由树



## 设计一个高并发网络，你选 TCP 还是 UDP？

> 面对高并发场景，真正决定决定选择的是**业务数据一致性和延迟的容忍程度**
>
> * 选择TCP场景
>
>   * 核心诉求：绝对不能丢包，绝对不能乱序，宁可慢一点
>   * 应用场景：
>     * 支付金融系统：各种支付、转账对账，一分钱都不能出错，必须依赖TCP
>
> * 选择UDP场景
>
>   * 核心诉求：极致的低延迟，允许偶尔丢包，但绝对不能容忍前面的包丢了，导致后面的包全卡在内核
>   * 应用场景：
>     * 实时对战游戏：玩家的移动坐标、技能释放，采用UDP+应用层可靠性封装。稍微丢一帧，客户端可以通过其它方法平滑过去，但是绝不能卡顿（某真投入游戏服务器，点你呢）
>     * 实时音视频：视频会议、直播，偶尔卡顿一瞬间是可以接收的，但不能延迟3秒才听到声音
>
> * 怎么实现？
>
