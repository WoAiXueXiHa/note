# 大模型的基本认知

## langchain和大模型的关系

**langchain是连接AI模型和实际AI应用的桥梁**

比如：

* 聊天类智能客服
* 数据分析助手 DS
* 豆包、Kimi、ChatGPT这类生成式应用
* 企业内部知识库问答
* 自动写报告、自动查数据、自动调用工具的Agent

直接使用AI模型，也就是直接调用原生API，会存在一些问题：

```text
1. 只能单轮对话，不好管理上下文
2. 不好接入外部知识库
3. 不好调用工具，比如查数据库、调用接口、搜索网页
4. 不好把多个步骤串起来
5. 不好管理提示词模板
6. 不好管理不同模型之间的切换
7. 不好做复杂业务流程
```

我目前的认知：

**langchain是把底层的大模型API进行了封装和改造，让大模型从一个单纯的聊天接口，变成一个可以接入工具、记忆、知识库、业务流程的应用开发框架 **

LangChain不是大模型本身 

LangChain更像是一个“应用层框架” 

大模型负责：

```text
理解语言
生成内容
进行推理
做出判断
```

LangChain负责：

```text
管理提示词
管理上下文
连接知识库
调用外部工具
编排多个步骤
把模型能力接入业务系统
```

所以可以理解为：

```text
大模型 = 大脑
LangChain = 神经系统 + 工具箱 + 工作流编排器
业务系统 = 身体和现实环境
```

## 宏观大模型认知

大模型是一个超级牛逼的博士，数字巨人 

它读过大量书，见过大量代码，知道很多知识，能写文章，能写代码，能分析问题 

但是大模型本身也有局限：

```text
1. 它不知道企业内部的私有数据
2. 它不会主动操作真实系统
3. 它不知道当前最新的信息
4. 它可能一本正经地胡说
5. 它不能天然融入业务流程
```

所以大模型的核心问题是什么？

**原生大模型不能融入业务流程 **

更准确地说：

```text
大模型本身只是一个语言预测器 
它擅长理解和生成语言 
但是它不能天然知道你的数据库在哪里
不能天然调用你的订单接口
不能天然读取你的公司文档
也不能天然完成一个完整的业务闭环 
```

所以真正的问题是：

**怎么让大模型不仅仅是一个百科全书，而是成为一个能行动、思考、融入业务流程的智能伙伴，也就是Agent？**

LangChain就是为了让用户更好使用AI模型这个牛逼的大脑 

它把大模型从：

```text
只能回答问题的聊天机器人
```

升级为：

```text
能读资料、能用工具、能记忆上下文、能执行任务的智能应用 
```

# 大模型

**模型：从数据中学习规律的数学函数或者程序 **

它本质上是在模仿某种规律 

比如：

```text
[2, 4, 8]  -> 8
[3, 4, 14] -> 14
[20, 10, 30] -> 30
```

如果模型从这些数据里找到规律：

```text
输出三个数中的第三个数
```

那它下次看到：

```text
[7, 9, 100]
```

就会输出：

```text
100
```

所以模型不是靠人一条一条写规则，而是从大量数据里自己学规律 

模型可以理解为：

```text
输入 -> 模型 -> 输出
```

比如：

```text
图片 -> 图像识别模型 -> 猫 / 狗
文本 -> 情感分析模型 -> 正面 / 负面
用户问题 -> 大语言模型 -> 回答
用户行为 -> 推荐模型 -> 推荐商品
```

## 使用方法

使用一个模型，一般分为两个阶段：

```text
1. 训练阶段
2. 推理阶段
```

### 训练阶段

训练阶段就是让模型学习规律 

```text
大量数据 + 标准答案 -> 模型不断调整参数 -> 学到规律
```

比如训练猫狗识别模型：

```text
图片A -> 猫
图片B -> 狗
图片C -> 猫
图片D -> 狗
```

模型不断看这些图片，然后调整内部参数，最后学会区分猫和狗 

### 推理阶段

推理阶段就是使用训练好的模型 

```text
新输入 -> 已训练模型 -> 输出结果
```

比如：

```text
一张新图片 -> 模型 -> 判断是猫
```

在大语言模型中：

```text
用户输入Prompt -> LLM -> 输出回答
```

## 使用场景

模型可以用在很多地方：

```text
1. 图像识别
2. 语音识别
3. 文本分类
4. 推荐系统
5. 搜索排序
6. 风控判断
7. 自动驾驶
8. 大语言模型问答
```

简单来说：

**只要一个问题可以从历史数据中学习规律，就可以尝试使用模型解决 **

## 模型关键特点

### 1. 一个模型通常专精一件事情

传统模型一般比较专一 

比如：

```text
垃圾邮件分类模型：判断是不是垃圾邮件
房价预测模型：预测房价
图像识别模型：识别图片内容
推荐模型：推荐用户可能喜欢的东西
```

它们不像大语言模型这么通用 

### 2. 需要标注原始数据

传统模型通常需要大量标准答案 

比如：

```text
这张图是猫
这张图是狗
这句话是正面评价
这句话是负面评价
```

模型根据标准答案来调整自己 

### 3. 参数较少

参数可以理解为：**模型从数据中学到的规则 **

参数越多，模型理论上能表达的规律越复杂，能力也越强 

但是参数多也意味着：

```text
1. 训练成本更高
2. 推理成本更高
3. 需要更多数据
4. 部署更困难
```

# 大语言模型 LLM

LLM，全称是 Large Language Model 

中文叫：**大语言模型 **

定义：**基于大规模神经网络，通过自监督或者半监督方式，对海量文本进行训练的语言模型 **

拆开理解：

```text
大：参数规模大，训练数据大，计算资源大
语言：主要处理自然语言
模型：从数据中学习规律的程序
```

所以LLM可以理解为：**一个超级自动补全系统 **

你输入一段话，它预测后面最可能出现什么内容 

比如：

```text
我今天中午想吃
```

它可能补全：

```text
一碗牛肉面 
```

再比如：

```text
请帮我写一个快速排序
```

它可能补全出一段代码 

表面上看它是在“回答问题”，本质上是它根据训练中学到的语言、知识、代码、逻辑模式，预测并生成最合理的后续内容 

## 神经网络

神经网络可以理解为：**一个极其高效的条件反射链，模仿人脑的工作方式 **

它由大量虚拟的神经元和连接组成 

每个神经元都是一个小处理单元，负责处理一点点信息 

```text
前一层的输出 -> 后一层的输入
```

一层一层传递，最后得到结果 

训练神经网络时，需要大量数据：

```text
输入数据 -> 神经网络 -> 输出结果 -> 和标准答案比较 -> 调整参数
```

这个过程会重复很多次 

每次训练，模型都会稍微调整内部参数 

最后形成一个复杂的判断流水线 

神经网络适合处理复杂问题：

```text
1. 图像识别
2. 语音识别
3. 机器翻译
4. 文本生成
5. 自动驾驶
6. 推荐系统
7. 大语言模型
```

传统程序适合规则清晰的问题 

神经网络适合规则很难手写，但是可以通过大量数据学习的问题 

## 自监督

自监督可以理解为：**自己从数据中制造标准答案 **

最典型的例子就是：

```text
完形填空
```

比如一句话：

```text
今天北京的天气很____ 
```

模型需要预测空白处是什么 

答案可能是：

```text
好
热
冷
晴
```

对于大语言模型来说，海量文本本身就是训练材料 

不需要人工一条一条标注 

因为文本天然有顺序：

```text
前面的词 -> 后面的词
```

模型可以通过预测下一个词来学习语言规律 

训练时可以这样：

```text
输入：我今天去了
目标：公园

输入：人工智能正在
目标：改变世界

输入：请帮我写一个
目标：函数
```

模型不断做这种“预测下一个词”的任务 

做得多了，它就学到了：

```text
语言结构
常识知识
语法规则
代码模式
推理模式
表达风格
```

自监督适合海量无标注数据 

比如：

```text
1. 网页文本
2. 书籍
3. 论文
4. 代码仓库
5. 新闻
6. 百科
7. 对话数据
```

大语言模型主要就是靠自监督训练出来的 

## 半监督

半监督可以理解为：**师傅领进门，修行在个人 **

一部分数据有标准答案，一部分数据没有标准答案 

模型先通过少量有标签数据学到方向，再利用大量无标签数据继续学习 

比如情感分类：

有标签数据：

```text
这个电影太好看了 -> 正面
这个商品质量很差 -> 负面
```

无标签数据：

```text
这家店我下次还会来
这个手机用起来一般
```

模型先从有标签数据中学会基本判断，再从无标签数据中继续学习语言分布 

半监督适合：

```text
1. 标注数据代价搞
2. 无标注数据很多
3. 希望模型有更强泛化能力
4. 传统监督学习效果不够好
```

## 语言模型

语言模型就是：**预测语言出现概率的模型 **

简单说：**它判断一句话后面最可能接什么 **

比如：

```text
我爱吃苹果，因为苹果很
```

后面可能是：

```text
甜
好吃
健康
新鲜
```

语言模型会给这些候选词一个概率 

```text
甜：40%
好吃：30%
健康：20%
新鲜：10%
```

然后选择合适的词输出 

在使用LLM时，人给它一个Prompt 

```text
Prompt -> 语言模型 -> Completion
```

比如：

```text
Prompt：请解释一下什么是递归
Completion：递归是一种函数调用自身的编程思想...
```

现在的聊天式大模型也是这个逻辑，只是外面包了一层对话格式 

```text
用户消息 -> 模型理解上下文 -> 生成助手回复
```

语言模型适合：

```text
1. 文本生成
2. 文本总结
3. 翻译
4. 问答
5. 代码生成
6. 对话机器人
7. 信息抽取
8. 文档改写
```

LLM首先是语言大师 

它可以理解自然语言，也可以生成自然语言 

比如：

```text
写文章
写邮件
写总结
改写句子
翻译文本
提取重点
```

使用时要给清楚任务 

比如：

```text
请把下面这段话总结成3个要点 
请用更正式的语气改写这封邮件 
请把下面的中文翻译成英文 
```

具体的使用场景：

```text
1. 写作助手
2. 文档总结
3. 邮件生成
4. 翻译
5. 内容运营
6. 客服话术生成
```

## 2. 知识巨人，百科全书

LLM在训练中看过海量文本，所以它储存了大量通用知识 

它像一个知识面极广的博士 

但是要注意：

```text
它不是数据库 
它记住的是概率模式，不是精确事实表 
```

所以它可能会出现幻觉 

幻觉就是：

**模型看起来很自信，但说的是错的 **

如果问通用知识，可以直接问 

```text
什么是操作系统？
什么是TCP三次握手？
什么是二叉树？
```

如果问精确信息，最好接入外部资料 

```text
公司制度
最新新闻
实时股价
数据库数据
用户订单信息
```

这类问题最好通过：

```text
搜索引擎
数据库查询
工具调用
```

来增强 

使用场景：

```text
1. 学习助手
2. 概念解释
3. 知识问答
4. 培训材料生成
5. 技术文档辅助理解
```

---

## 3. 逻辑和代码，从思维到实现

LLM可以进行一定程度的逻辑推理和代码生成 

它不只是会说话，还会把需求转换成步骤 

比如：

```text
分析题意
拆解任务
生成代码
解释报错
优化 SQL
写测试用例
```

问代码问题时，要给足上下文：

```text
1. 使用什么语言
2. 当前代码是什么
3. 报错信息是什么
4. 期望结果是什么
5. 实际结果是什么
```

好的问法：

```text
我用C++写二分查找，下面代码在边界用例出错 
请帮我指出bug，并解释为什么 
```

不好的问法：

```text
我代码错了，帮我看看 
```

```text
1. 编程学习
2. 代码生成
3. Bug修复
4. 代码解释
5. 算法题辅助
6. SQL生成
7. 单元测试生成
```

比如：

文字
图片
语音
表格
视频

# 提示词编写

提示词的核心是换位思考 

你不能假设模型天然知道你想要什么 

你要告诉它：

```text
你是谁
你要做什么
背景是什么
目标是什么
限制是什么
输出什么格式
```

一句话：

**提示词就是给大模型写任务说明书 **

---

## 结构化框架

结构化提示词就是把需求拆成清晰的模块，而不是随便说一句话 

一个好的提示词通常包含：

```text
1. 任务背景和上下文
2. 核心目标
3. 执行步骤
4. 语言风格
5. 目标用户
6. 输出格式
7. 约束条件
8. 示例
```

可以按这个模板写：

```text
角色：
你是一个资深C++后端工程师 

背景：
我正在学习epoll多路转接，目前不理解为什么要设计epoll，它比select/poll好在哪里

目标：
请你用初学者能听懂的方式解释我的疑问

要求：
1. 先给定义
2. 再用生活例子解释
3. 再给代码例子
4. 最后总结区别

输出格式：
使用Markdown，分点说明 
```

结构化提示词适合：

```text
1. 学习新概念
2. 生成文档
3. 写代码
4. 写方案
5. 做总结
6. 做复杂任务
```

越复杂的任务，越需要结构化提示词 

## 少样本提示、多示例提示

少样本提示，也叫Few-shot Prompting 

核心思想：**不下达复杂指令，而是让模型照葫芦画瓢 **

你给它几个例子，它就会模仿这些例子的格式、风格和逻辑 

比如做数据提取：

```text
示例1：
输入：张三，手机号13811112222，住在北京
输出：
姓名：张三
手机号：13811112222
城市：北京

示例2：
输入：李四，电话13933334444，来自上海
输出：
姓名：李四
手机号：13933334444
城市：上海

现在处理：
输入：王五，手机号13688889999，住在杭州
输出：
```

模型看到前面的模式，就会自动生成：

```text
姓名：王五
手机号：13688889999
城市：杭州
```

适用于：

```text
1. 风格仿写
2. 数据提取
3. 固定格式生成
4. 分类任务
5. 复杂格式转换
6. 逻辑模式学习
```

比如：

```text
把自然语言转换成SQL
把用户评价分类
把文本转换成JSON
模仿某种写作风格
```

## 思维链提示

### 定义

思维链提示，也叫Chain of Thought 

核心思想：**让模型不要直接给答案，而是先拆解问题，再一步一步推理 **

**当好的提示词能帮助模型解决原来解决不了的难题，说明提示词提升了模型的推理水平 **

常见写法：

```text
请一步一步进行分析，并得出结论 
```

或者：

```text
请先拆解问题，再逐步推理，最后给出答案 
```

少样本思维链：

```text
问题：小明有3个苹果，又买了2个，一共有几个？
思考：原来有3个，又增加2个，所以是3+2=5 
答案：5 

问题：小红有10元，买笔花了4元，还剩多少？
思考：
```

零样本思维链：

```text
请一步一步进行推理，并得出结论 
```

适合：

```text
1. 数学题
2. 算法题
3. 逻辑推理
4. 多步骤任务
5. 复杂决策
6. 错误分析
```

比如：

```text
请一步一步分析这段代码为什么会死循环 
请一步一步推导这个SQL为什么查不出数据 
请一步一步分析这个算法的时间复杂度 
```

注意：**思维链不是万能的** 

模型可能会：

```text
1. 推理过程看起来合理，但结论错
2. 为了圆答案编造过程
3. 在复杂问题上中途跑偏
```

所以复杂任务最好加上：

```text
请检查你的推理过程是否有漏洞 
请给出最终答案，并说明不确定的地方 
```

## 自我批判和迭代

### 定义

自我批判就是：**把生成和评审两个步骤分离 **

不要让模型一次性既生成又保证完美 

而是先让它生成初稿，再让它站在评审者角度检查问题，最后修改 

可以分三步：

```text
第一步：生成初稿
第二步：审查问题
第三步：根据问题重新优化
```

示例：

# LLM接入方式

## API接入

### 流程

API接入就是：**通过HTTP接口调用云端大模型 **

比如：

```text
OpenAI API
通义千问 API
智谱 API
DeepSeek API
豆包 API
Claude API
Gemini API
```

你不需要自己部署模型，只需要把请求发给模型服务商 

简单流程：

```text
用户输入 -> 后端服务 -> 大模型API -> 返回结果 -> 展示给用户
```

一般需要：

```text
1. 注册模型平台账号
2. 获取API Key
3. 选择模型
4. 组装请求参数
5. 发送HTTP请求
6. 解析模型返回结果
```

常见参数：

```text
model：使用哪个模型
messages：对话内容
temperature：控制随机性
max_tokens：控制最大输出长度
stream：是否流式输出
```

适合：

```text
1. 快速开发AI应用
2. 企业内部系统接入
3. 智能客服
4. AI写作
5. 代码助手
6. 知识库问答
7. 不想自己部署模型的场景
```

### 优点

```text
1. 接入快
2. 不用买显卡
3. 模型能力强
4. 维护成本低
5. 可以直接用最新模型
```

### 缺点

```text
1. 需要联网
2. 数据会发给第三方
3. 长期调用有成本
4. 受限于厂商接口
5. 私有化程度不高
```

---

## 本地部署

### 流程

本地部署就是：**把大模型下载到自己的机器或者服务器上运行 **

比如使用：

```text
Ollama
vLLM
LM Studio
llama.cpp
FastChat
Xinference
```

可以部署的开源模型：

```text
Llama
Qwen
DeepSeek
ChatGLM
Baichuan
InternLM
Mistral
```

一般流程：

```text
1. 选择开源模型
2. 准备机器资源
3. 下载模型权重
4. 使用推理框架启动模型服务
5. 通过本地API调用模型
```

比如使用Ollama：

```text
安装Ollama
下载模型
启动模型
通过本地接口调用
```

适合：

```text
1. 数据隐私要求高
2. 公司内部系统
3. 离线环境
4. 学习和实验
5. 边缘设备
6. 对成本可控性要求高
```

### 优点

```text
1. 数据不出本地
2. 可控性强
3. 可以私有化部署
4. 可以做微调
5. 长期成本可能更低
```

### 缺点

```text
1. 需要硬件资源
2. 部署复杂
3. 模型能力可能不如顶级闭源模型
4. 运维成本高
5. 推理速度受机器影响
```

## SDK接入

SDK接入就是：**模型厂商或者框架提供一套封装好的开发工具包，让开发者更方便地调用模型 **

API是更底层的HTTP接口 

SDK是在API上封装了一层 

比如：

```text
Python SDK
Java SDK
JavaScript SDK
Go SDK
```

一般流程：

```text
1. 安装SDK
2. 配置API Key
3. 初始化客户端
4. 调用模型方法
5. 获取返回结果
```

比如伪代码：

```python
client = ModelClient(api_key="你的key")

response = client.chat(
    model="xxx",
    messages=[
        {"role": "user", "content": "解释一下什么是LLM"}
    ]
)

print(response)
```

适合：

```text
1. 正式项目开发
2. 后端服务集成
3. Python脚本调用
4. 快速搭建Demo
5. 和业务代码结合
```

### API和SDK的区别

```text
API：
更底层，直接发HTTP请求，自由度高，但代码多 

SDK：
封装更好，调用方便，适合快速开发，但受SDK设计限制 
```

可以这样理解：

```text
API = 原材料
SDK = 半成品工具包
LangChain = 应用开发框架
Agent = 能自主使用工具完成任务的智能体
```
