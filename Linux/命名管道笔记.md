## 一、先解答核心问题：shmget 的 key 为什么要用户传入？

`key` 要求用户传入的根本目的是 **为了让多个不相关进程能够找到并访问同一个内核级共享内存段，实现跨进程的内存共享**，具体可从以下 4 个层面理解：

### 1. 共享内存的核心需求：跨进程关联同一资源

`shmget` 是 System V 共享内存的核心函数，其创建 / 获取的共享内存段是 **内核级资源**（生命周期与进程无关，只与内核生命周期或手动删除相关），而非进程私有资源。

对于父子进程，可通过继承匿名内存映射（如 `mmap` 的 `MAP_ANONYMOUS` 标志）实现共享，无需全局标识；但对于 **不相关进程**（如两个独立启动的应用程序），它们没有继承关系，无法直接感知对方创建的资源，此时就需要一个 **全局唯一的 “资源标识”** 来让它们关联到同一个共享内存段，这个标识就是 `key`。

如果 `key` 由系统自动分配，每个进程调用 `shmget` 得到的 `key` 会不同，无法实现 “找到同一个共享内存段” 的核心需求，因此必须由用户（开发者）传入一个统一的 `key`（或通过 `ftok` 生成统一 `key`），让需要通信的多个进程使用相同 `key` 调用 `shmget`。

### 2. key 的本质：内核级 IPC 资源的 “全局身份证”

System V 系列 IPC 资源（共享内存、消息队列、信号量）都依赖 `key_t` 类型的 `key` 作为全局标识：

* 内核内部会维护一张 IPC 资源表，每个已创建的 System V IPC 资源都会在表中占据一条记录，`key` 是这条记录的 “主键”，保证全局唯一（同一系统中，同一个 `key` 只能对应一个同类型的 IPC 资源）。
* 用户传入 `key` 后，内核会根据该 `key` 检索 IPC 资源表：存在则返回该资源的句柄，不存在则根据 `shmflg` 标志决定是否创建新资源。

### 3. 用户传入 key 的灵活性：两种常用方式

用户传入 `key` 并非只能硬编码一个数字，而是有两种更合理的方式，兼顾唯一性和可维护性：

* 方式 1：硬编码统一常量（如 `#define SHM_KEY 0x12345678`），适用于简单场景，需要确保不同进程使用相同常量。
* 方式 2：通过 `ftok()` 函数生成（推荐），原型 `key_t ftok(const char *pathname, int proj_id)`，它将一个 **存在且可访问的文件路径** 和一个 **自定义项目 ID（1-255）** 转换为唯一 `key`。只要多个进程使用相同的 `pathname` 和 `proj_id`，就能生成相同的 `key`，同时避免了硬编码的冲突问题。

### 4. 特殊例外：IPC_PRIVATE（无需用户自定义 key）

`shmget` 支持一个特殊的 `key` 值：`IPC_PRIVATE`（本质是 0），此时无需用户传入自定义全局 `key`。但它的作用是创建一个 **仅对父子进程可见的私有共享内存段**，无法用于不相关进程之间的通信，这也从反面印证了：**要实现不相关进程的共享，必须由用户传入全局唯一的 `key`**。

**并发编程**

* 多个执行流（进程），能看到同一份公共资源：共享资源

* 被保护起来的资源：临界资源

* 保护方式：互斥、同步

* 系统中某些资源一次只允许一个进程使用：临界资源or互斥资源（任何时刻，只运行一个执行流访问资源）

  同步：让多个执行流在访问临界资源时候，有一定的顺序性

* 代码访问临界资源的这部分代码->临界区

  代码=临界区代码+非临界区代码
