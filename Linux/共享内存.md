## 共享内存示意图

![image-20260122095407408](https://gitee.com/binary-whispers/pic/raw/master///20260122095409502.png)

**什么是共享内存？**

OS直接将同一块物理内存映射到两个不同进程的虚拟地址空间

**有什么优势？**

最快的IPC，数据零拷贝，一旦映射成功，进程就像访问自己的堆栈一样访问这块内存，不需要执行进入内核的系统调用来传递信息，执行系统调用要经过**用户空间->内核空间->用户空间**两次拷贝

## 共享内存函数

**创建共享内存**

>`shmget`
>
>```cpp
>   #include <sys/shm.h>
>
>   int shmget(key_t key, size_t size, int shmflg);
>```
>
>`key`：共享内存段名字
>
>`size`：共享内存大小
>
>`shmflg`：用法和创建文件的`mode`一样
>
>* `IPC_CREAT`：共享内存不存在，创建并返回；共享内存存在，获取并返回->用于获取
>* `IPC_CREAT | IPC_EXCL`：共享内存不存在，创建并返回；共享内存已存在，出错返回->用于创建
>
>返回值：成功返回共享内存段标识符，失败返回-1

**问题：key为什么要人为传入，怎么传入？**

> * 人为传入原因：
>
>   一个进程A创建了一块共享内存，一个进程B想要用**同一块**内存
>
>   A和B是两个相互独立的进程，**二者虚拟地址隔离**，OS随机分配给A一个ID，B根本不知道这个ID是多少（除非A通过某种方式告诉B，但是这就陷入了“为了通信而通信”的死循环）
>
> * 怎么传入：
>
>   ```cpp
>   #include <sys/ipc.h>
>   
>   key_t ftok(const char *pathname, int proj_id);
>   ```
>
>   **`pathname` (路径名)**：必须是一个**已经存在**且**可访问**（至少可见）的文件路径
>
>   **`proj_id` (项目 ID)**：实际上**只有低 8 位有效**（即范围 0-255，或者是字符 'a', 'A' 等）

**问题：key和shmid的区别？**

> * key类似文件路径，是`ftok`生成或认为指定的外部标识符，用来查找匹配资源->找内存
> * shmid类似文件描述符，由内核分配，是内部操作的标识->用内存

**挂接共享内存->连接到进程地址空间**

> `shmat`
>
> ```cpp
>    #include <sys/shm.h>
> 
>    void *shmat(int shmid, const void *_Nullable shmaddr, int shmflg);
> 
> ```
>
> `shmid`：共享内存标识符
>
> `shmaddr`：指定连接的地址
>
> `shmflg`：`SHM_RND`地址对齐 `SHM_RDONLY`只读权限
>
> 返回值：成功返回指向共享内存第一个节的指针，失败返回-1
>
> 说明： 
>
> * `shmaddr`为空，自动选择一个地址
> * `shmaddr`不为空&&`shmflg`无`SHM_RND`标记，`shmaddr`为连接地址
> * `shmaddr`不为空&&`shmflg`设置了`SHM_RND`标记，连接的地址向下取整，调整为`SHMLBA`的整数倍

**去关联->将共享内存段和进程脱离**

>`shmdt`
>
>```cpp
>   #include <sys/shm.h>
>
>   int shmdt(const void *shmaddr);
>```
>
>注意：去关联不等于删除共享内存段

**控制共享内存**

>`shmctl`
>
>```cpp
>   #include <sys/shm.h>
>
>   int shmctl(int shmid, int op, struct shmid_ds *buf);
>```
>
>`shmid`：`shmget`返回的共享内存标识符
>
>`op`：采取的动作
>
>* `IPC_STAT`：状态查询，把内核中关于这个共享内存的信息，拷贝到`buf`里
>
>* `IPC_SET`：修改共享内存的属性，普通用户只能修改权限，不能修改大小
>
>* `IPC_RMID`：标记删除这块共享内存
>
>  注意：
>
>  * 如果调用了 `IPC_RMID`，但如果此时还有进程挂接在这块内存上，物理内存**不会**立刻释放
>  * 内核会将 `shmid_ds` 结构中的 `mode` 字段标记为 `SHM_DEST`->新进程无法找到这块内存，指导最后一个进程去关联后，引用计数变为0，内核才回收物理内存

**内核如何管理共享内存？**

```cpp
struct shmid_ds {
    struct ipc_perm shm_perm;    /* 权限结构体 (uid, gid, mode) */
    size_t          shm_segsz;   /* 共享内存大小 (bytes) */
    time_t          shm_atime;   /* 最后一次 attach 时间 */
    time_t          shm_dtime;   /* 最后一次 detach 时间 */
    time_t          shm_ctime;   /* 最后一次 change 时间 */
    pid_t           shm_cpid;    /* 创建者 PID */
    pid_t           shm_lpid;    /* 最后操作者 PID */
    shmatt_t        shm_nattch;  /* 当前挂接（Attach）的进程数量 */
    // ... 其他保留字段
};
```

`shm_nattch`是内核判断是否真正释放内存的计数器

`IPC_RMID` 只是将内存段标记为销毁状态，并移除 Key，防止新进程连接。但真正的物理内存释放，要等到所有已连接的进程都执行 `shmdt`，即引用计数降为 0 时才会发生。这是一种保护机制，防止强制删除导致正在使用的进程崩溃

## 实战：利用共享内存实现通信，利用管道解决共享内存无法同步问题

### 核心架构设计

构建两个独立的进程：server、client，通过以下方式协作：

1. **公共契约：** 双方约定好共享内存的生成规则（路径+ID）和共享内存里放什么
2. **数据通道：** server建立仓库并收货，client发货
3. **控制通道：** 命名管道作为通信信道，client发货，通过管道传递消息之后，server知道后取货
