## 0. 你这套进程池到底做了什么（先把大图建立起来）

* **Master（父进程）**：创建 N 个 Worker（子进程），并给每个 Worker 建一条“控制管道”（父写子读）。
* **Worker（子进程）**：循环从管道读一个 `cmd`（int），把它当作任务编号，去任务表里执行对应函数。
* **任务派发**：Master 轮询选择一个 Worker，把 `cmd` 写入该 Worker 对应的管道写端。
* **退出**：Master 关闭所有写端 -> Worker 读到 EOF（`read` 返回 0）-> Worker 退出 -> Master `waitpid` 回收。

这就是经典的 “**command pipe + fork workers**” 模式。

### 1.1 `Channel.hpp`（封装“父 -> 子”的通信信道）

```cpp
#pragma once

#include <iostream>
#include <string>
#include <unistd.h>     // write/close
#include <sys/types.h>  // pid_t

// Channel：父进程持有“写端fd”，用于向某个子进程发送命令（cmd）
class Channel {
public:
    Channel(int wfd, pid_t who)
        : _wfd(wfd), _who(who)
    {
        // 仅用于debug输出：Channel-<wfd>-<pid>
        _name = "Channel-" + std::to_string(wfd) + "-" + std::to_string(who);
    }

    // 建议：这些 getter 可以标记为 const
    std::string getName() const { return _name; }
    int getWfd() const { return _wfd; }
    pid_t getId() const { return _who; }

    // 向子进程发送一个命令编号
    // 重要知识点：write 可能被信号中断(EINTR)，也可能部分写入（对 pipe 而言写 <= PIPE_BUF 通常是原子）
    void Send(int cmd) {
        // 在面试里你可以说：这属于“协议层”，目前协议是一条消息=一个int
        ssize_t n = ::write(_wfd, &cmd, sizeof(cmd));
        (void)n; // demo中不处理错误；工程里应处理EINTR/EPIPE等
    }

    // 关闭父进程持有的写端：这是“通知子进程退出”的关键动作
    void Close() {
        if (_wfd >= 0) {
            ::close(_wfd);
            _wfd = -1;  // 防止重复close（重复close是很常见坑点）
        }
    }

    ~Channel() {
        // demo里不自动close，避免对象拷贝/移动导致重复close的复杂性
        // 工程里一般做 RAII：析构 close，但必须保证“不可拷贝/可移动”语义正确
    }

private:
    int _wfd;            // 父写端fd
    std::string _name;   // debug名字
    pid_t _who;          // 对应子进程pid
};
```

**面试加分点**：你可以主动提到

* `Channel` 的析构是否应该 `close`？如果做 RAII，就要把 `Channel` 设为 **不可拷贝**（delete copy ctor/assign），只允许移动，否则 vector 扩容拷贝会导致双重 close。你现在用 `emplace_back` + 空析构规避了这一类雷（但也引入资源泄露风险，靠 `CleanProcessPool` 来集中 close）。

### 1.2 `Task.hpp`（任务表 + Worker）

```cpp
#pragma once

#include <iostream>
#include <vector>
#include <functional>
#include <ctime>
#include <unistd.h>

// task_t：一个“可调用对象”的统一包装类型：能被()调用，签名为 void()
using task_t = std::function<void()>;

// TaskManger：维护一个任务列表，每个任务是一个可调用对象
class TaskManger {
public:
    TaskManger() {
        // 注意：rand/srand 是全局状态，且非线程安全（多线程会有问题）。
        // 你这里是多进程，每个子进程fork后会复制父进程状态，
        // 如果在fork前只srand一次，子进程可能产生相同随机序列（面试常问）。
        srand(time(nullptr));

        // 这里用 lambda 生成可调用对象，捕获列表为空[]，所以它们是无状态函数对象
        tasks.push_back([]() {
            std::cout << "sub process[" << getpid() << "]访问数据库\n\n";
        });
        tasks.push_back([]() {
            std::cout << "sub process[" << getpid() << "]url解析\n\n";
        });
        tasks.push_back([]() {
            std::cout << "sub process[" << getpid() << "]加密任务\n\n";
        });
        tasks.push_back([]() {
            std::cout << "sub process[" << getpid() << "]数据持久化\n\n";
        });
    }

    int SelectTask() { return rand() % tasks.size(); }

    // Execute：执行第 number 个任务
    void Execute(size_t number) {
        // 工程里建议做边界检查；demo略
        tasks[number]();
    }

private:
    std::vector<task_t> tasks;
};

// 全局任务管理器：
// 注意：全局对象初始化顺序在多文件中可能踩坑（静态初始化顺序问题）。
// 建议工程化写法：放到cpp里，用函数返回单例（Meyers singleton）或由main构造并传入。
TaskManger tm;

// Worker：子进程核心循环
inline void Worker() {
    while (true) {
        int cmd = 0;

        // 读标准输入fd=0；因为子进程里把pipe读端dup2到了0
        ssize_t n = ::read(0, &cmd, sizeof(cmd));

        if (n == sizeof(cmd)) {
            // 成功读到一个完整命令
            tm.Execute(cmd);
        } else if (n == 0) {
            // read返回0：对端全部关闭 => 管道写端全部关闭 => EOF
            // 这就是“父进程关闭写端通知子进程退出”的机制
            std::cout << "pid: " << getpid() << " quit ...\n";
            break;
        } else {
            // n < 0：错误；工程里应处理 EINTR（被信号中断重试）等
            // demo略过
        }
    }
}
```

### 1.3 `ProcessPool.hpp`（创建子进程、建立信道、派发任务、清理）

```cpp
#pragma once

#include <iostream>
#include <string>
#include <vector>
#include <cstdlib>
#include <unistd.h>
#include <sys/types.h>
#include <sys/wait.h>
#include <functional>

#include "Task.hpp"
#include "Channel.hpp"

using work_t = std::function<void()>;

// 错误码
enum {
    OK = 0,
    UsageError,
    PipeError,
    ForkError
};

// ProcessPool：
// 1) Init：创建N个子进程，每个子进程有自己的“控制管道”
// 2) Dispatch：父进程向不同子进程写cmd
// 3) Clean：关闭fd并wait回收子进程
class ProcessPool {
public:
    ProcessPool(int n, work_t w)
        : _processNum(n), _work(std::move(w)) {}

    int InitProcessPool() {
        for (int i = 0; i < _processNum; i++) {
            int pipefd[2] = {0, 0};

            // pipefd[0]：读端；pipefd[1]：写端
            // 我们要实现“父写子读”
            if (pipe(pipefd) < 0) return PipeError;

            pid_t pid = fork();
            if (pid < 0) return ForkError;

            if (pid == 0) {
                // ---------------- 子进程分支 ----------------

                // 面试要点：子进程继承父进程所有已打开fd！
                // 所以子进程必须关闭“历史管道写端”，否则：
                // 1) fd泄露
                // 2) 影响EOF语义：如果某子进程还持有某管道写端，则对应读端永远不会收到EOF
                std::cout << getpid() << ", child close history fd: ";
                for (auto &c : _channels) {
                    std::cout << c.getWfd() << " ";
                    c.Close();
                }
                std::cout << " over\n";

                // 子进程不需要写端，关闭pipefd[1]
                ::close(pipefd[1]);

                // 把pipe读端dup到stdin(fd=0)，Worker里直接read(0,...)即可
                // 面试要点：dup2 会先close目标fd(0)再复制
                dup2(pipefd[0], 0);

                // dup2之后，原始pipefd[0]也可以关闭（避免多余fd）
                ::close(pipefd[0]);

                // 执行子进程工作函数（回调）
                _work();

                ::exit(0);
            }

            // ---------------- 父进程分支 ----------------

            // 父进程不需要读端，关闭pipefd[0]
            ::close(pipefd[0]);

            // 记录“写端 + 子进程pid”为一个Channel
            // emplace_back 直接原地构造 Channel，减少临时对象
            _channels.emplace_back(pipefd[1], pid);
        }

        return OK;
    }

    void DispatchTask() {
        int who = 0;
        int num = 20;

        while (num--) {
            // 选择一个任务编号
            int task = tm.SelectTask();

            // 轮询选择一个子进程Channel
            Channel &cur = _channels[who++];
            who %= _channels.size();

            std::cout << "########################\n";
            std::cout << "send " << task << " to " << cur.getName()
                      << ", 任务剩余数：" << num << "\n";
            std::cout << "########################\n";

            // 写cmd到对应子进程的管道
            cur.Send(task);

            sleep(1);
        }
    }

    void CleanProcessPool() {
        // 核心目标：
        // 1) 关闭所有写端 => 子进程read返回0 => 子进程退出
        // 2) waitpid回收所有子进程，避免僵尸进程

        // version3：你当前使用的写法（推荐）
        for (auto &c : _channels) {
            c.Close(); // 先关闭写端，触发对应子进程EOF退出
            pid_t rid = ::waitpid(c.getId(), nullptr, 0); // 阻塞等待该子进程退出
            if (rid > 0) {
                std::cout << "child " << rid << " wait... success\n";
            }
        }

        // 说明：为什么“close + waitpid”放在同一轮里也能行？
        // 因为每个子进程只依赖它自己的管道写端关闭即可退出，
        // 你这里对每个子进程逐个close其写端，然后wait它，逻辑成立。

        // 但工程里更常见的做法是：
        // 1) 先close所有channel（保证所有子进程都收到EOF）
        // 2) 再wait所有子进程（更稳健，避免某些复杂情形下互相影响）

        /*
        // 推荐工程化写法（更“强保证”）：
        for (auto &c : _channels) c.Close();
        for (auto &c : _channels) {
            pid_t rid = ::waitpid(c.getId(), nullptr, 0);
            if (rid > 0) std::cout << "child " << rid << " wait... success\n";
        }
        */

        // version2 / version1 你注释掉的，我下面会专门讲“坑点与差异”
    }

    void DebugPrint() const {
        for (const auto &c : _channels) {
            std::cout << c.getName() << "\n";
        }
    }

private:
    std::vector<Channel> _channels;
    int _processNum;
    work_t _work;
};
```

## 2) 关键机制与“管道重新理解”（面试要讲得很硬）

### 2.1 匿名管道的本质

* `pipe(fd)` 创建一个**内核缓冲区**，返回两个文件描述符：
  * `fd[0]`：读端
  * `fd[1]`：写端
* 它是**单向**的：写端写入的数据只能从读端读出。
* **阻塞语义**（默认阻塞）：
  * 读：缓冲区空 -> `read` 阻塞，直到有数据或对端全关
  * 写：缓冲区满 -> `write` 阻塞，直到有空间或读端全关（可能 SIGPIPE/EPIPE）

### 2.2 fork 后 fd 的继承（最核心）

* `fork()` 会把父进程的**文件描述符表**复制给子进程：
  **父子都持有同一个管道对象的引用**（指向同一内核 pipe buffer）。
* 这带来两个常见后果（坑点）：
  1. **fd 泄露**：不关掉不需要的端，fd 数量会持续增长
  2. **EOF 失效**：读端要读到 EOF，必须满足：**所有写端都关闭**。
     只要还有任何一个进程持有该管道的写端，读端就永远不会返回 0。

> 你在子进程里 `close history fd` 这段，就是在修复第 2 点：
> 防止子进程意外持有“其他子进程管道的写端”。

### 2.3 为什么要 `dup2(pipefd[0], 0)`

这是一种“重定向”技巧：把管道读端变成 stdin。这样 Worker 只需要读 fd=0。
面试可以这么说：

* 这是把“输入来源”抽象掉：Worker 只依赖标准输入，不关心具体来自 pipe 还是别的。
* `dup2` 会让 `0` 指向管道读端对应的内核对象。
* `dup2` 后应把原始 `pipefd[0]` close 掉，避免冗余 fd。

## 3) 必考坑点清单（你这份代码最该补的“工程化防雷”）

### 3.1 写入/读取的“协议”和原子性

你现在一条消息是 `int`，非常适合 pipe：

* 对 pipe 来说，**写入长度 <= PIPE_BUF** 时，write 是原子（不会和其他写交错）。
  但要注意：你是“一个子进程一个管道”，本就不会并发写同一管道，风险更低。
* 但是仍建议面试里提一句：真实工程要处理
  * `write` 被信号中断：`errno == EINTR` 重试
  * `read` 返回小于期望字节数：要循环读满（你这里读 int 通常一次读满，但规范上要考虑）

### 3.2 SIGPIPE / EPIPE

如果子进程提前退出，而父进程还往 pipe 写：

* 默认会触发 **SIGPIPE**，进程可能被杀掉
* 或 `write` 返回 -1，errno=EPIPE

工程常见做法：

* `signal(SIGPIPE, SIG_IGN)` 或者写时处理 EPIPE

### 3.3 全局 `tm`（静态初始化顺序问题）

`TaskManger tm;` 是全局对象，如果多个翻译单元里也有全局对象依赖它，可能出现“先用后建”。
更稳妥：

* 把 `tm` 放在一个 `.cpp` 并提供 `TaskManger& GetTM()`，函数内 static 生成。

### 3.4 Channel 的资源语义

你现在 `Channel` 析构不 close，依赖 `CleanProcessPool` 统一关。能跑，但工程面试可能追问：

* 如果异常提前返回，会不会泄露 fd？
* vector 扩容移动/拷贝会不会 double close？
  你可以回答：
* 正确工程做法：**RAII + 禁拷贝 + 可移动**，析构自动 close。

## 4) 清理进程池：三个版本逐层讲透（面试最爱问这里）

你注释里写了 v1 / v2 / v3。核心点只有一句：

> **要让子进程退出，必须让它读到 EOF；要读到 EOF，必须关闭所有写端；否则 waitpid 可能永远阻塞。**

下面逐个解释。

------

### 4.1 Version 1（你注释的 v1）

```cpp
for (auto &c : channels) {
    c.Close();
}
// 再 wait
for (auto &c : channels) {
    waitpid(c.Id(), ...);
}
```

**逻辑**：先把所有写端全关掉 -> 所有子进程几乎同时读到 EOF -> 都会退出 -> 再逐个 wait 回收。
**优点**：语义最清晰、最稳健（工程推荐）。
**面试表述**：这是“**广播退出信号**”再“**批量回收**”。

------

### 4.2 Version 2（你注释：倒序 wait，且说“阻塞了！”）

你写的是类似：

```cpp
for (int i = channels.size()-1; i >= 0; i--) {
    channels[i].Close();
    waitpid(channels[i].Id(), ...); // 阻塞了！
}
```

为什么可能阻塞？

* 如果你**只 close 当前这个子进程对应的写端**，但该子进程仍然“看见”其它写端没关（比如某些历史 fd 没关干净），它的 `read` 可能永远不返回 0，于是 `waitpid` 卡死。
* 这类问题往往来自：
  1. 子进程没有正确关闭历史 fd（你已经做了，这点很好）
  2. 父进程还有其它地方持有写端副本（比如 Channel 被拷贝、重复 fd）
  3. 子进程在执行任务时阻塞（例如任务中卡住），导致它迟迟不退出

**关键结论**：

* v2 是“边关边等”。它不是一定错，但**更脆弱**：任何一个子进程退出条件被破坏，都会让 master 卡住。
* v1 的“先全关再全等”能显著降低耦合与排查成本。

------

### 4.3 Version 3（你当前使用：边 close 边 wait，正序）

```cpp
for (auto& c : _channels){
    c.Close();
    waitpid(c.getId(), ...);
}
```

它和 v2 其实同一类（边关边等），为什么你现在这个看起来能工作？

* 你已经在子进程里做了“close history fd”操作，保障 EOF 语义更可靠。
* 你的 Worker 任务都很快，不太会卡住。
* 每个子进程只需要它自己的写端关闭就能读到 EOF（只要没有额外副本）。

**但面试里你最好给出更“工程化”的答法**：

* v3 在 demo 可行，但生产环境更推荐 v1 的“两阶段关停”。

> 建议你最终提交：**先 close 全部，再 wait 全部**。面试官通常会认可你知道这件事。🙂

## 5) C++11 语言点逐个讲清

### 5.1 `using task_t = std::function<void()>;` 是什么用法？

这是**类型别名**（typedef 的现代写法）：

* `std::function<void()>`：可包装任何“可调用对象”，只要它能以 `()` 调用且返回 `void`、无参数。
* 可调用对象包括：
  * 普通函数指针 `void foo()`
  * lambda `[](){...}`
  * 仿函数（重载 `operator()` 的类）
  * `std::bind` 生成的对象
* 作用：你把任务统一成一种类型，塞进 `vector<task_t>`，实现“任务表”。

面试经常追问两点：

1. `std::function` 有一定开销（类型擦除、可能堆分配），但换来巨大灵活性。
2. 如果追求极致性能，可用模板/函数指针/variant 等，但复杂度更高。

------

### 5.2 lambda 表达式怎么用（你这里属于最常见的无捕获 lambda）

你写的是：

```cpp
tasks.push_back([](){ std::cout << "..."; });
```

lambda 语法：

```cpp
[capture](params) mutable -> ret { body }
```

* `[]` 捕获列表：
  * `[]` 不捕获（无状态）
  * `[=]` 值捕获外部变量
  * `[&]` 引用捕获外部变量
  * `[x, &y]` 混合捕获
* `(params)` 参数列表
* `mutable` 允许修改“值捕获”的副本
* `-> ret` 返回类型（常可省略自动推导）

**无捕获 lambda**可以隐式转成函数指针（在某些场景），而捕获了变量的 lambda 不能转函数指针（它是带状态的闭包对象）。

------

### 5.3 什么是回调函数？你这里怎么实现回调？

“回调”本质：**把“要做什么”当作参数传进去**，由框架在合适时机调用它。

你这里：

```cpp
using work_t = std::function<void()>;
ProcessPool(int n, work_t w) : _work(w) {}
...
_work(); // 子进程里调用
```

* `ProcessPool` 是框架
* `Worker` 是业务逻辑
* `Worker` 作为回调传入，进程池不需要知道 Worker 内部细节

面试加分回答：

* 回调将“控制流”从调用者反转给框架（Inversion of Control, IoC）。
* `std::function` 让回调能接收 lambda/仿函数/成员函数绑定等多种形式。

------

### 5.4 C++11 范围 for（range-based for）怎么用

语法：

```cpp
for (auto& x : container) { ... }
```

* `auto&`：引用遍历，避免拷贝（推荐）
* `const auto&`：只读引用（更推荐只读场景）
* `auto`：会拷贝元素（对大对象不划算；对 Channel 还可能引发资源语义问题）

你这里必须用引用：

```cpp
for (auto& c : _channels) { c.Close(); }
```

如果写成 `auto c`（值拷贝），你关的是拷贝对象里的 fd（甚至导致逻辑错误）。

### 5.5 `emplace_back` vs `push_back` 区别，实践用哪个？

* `push_back(obj)`：把一个已经构造好的对象放进容器（拷贝/移动）
* `emplace_back(args...)`：在容器内部**原地构造对象**，避免临时对象

你这里：

```cpp
_channels.emplace_back(pipefd[1], pid);
```

优势：

* 直接调用 `Channel(int, pid_t)` 构造
* 少一次临时对象与移动（对资源对象更安全、更清晰）

**实践建议**：

* **优先 emplace_back**（尤其是资源/不可拷贝对象）
* 如果你已经有现成对象（比如 `Channel ch(...)`），`push_back(std::move(ch))` 也合理

## 1. 你现阶段已经掌握的核心知识点（按层级归纳）

### A. OS/进程模型层（你代码真正依赖的）

1. **fork 复制语义**

* `fork()` 后父子进程拥有**几乎完全相同的用户态地址空间快照**（写时复制 COW）。
* 更关键：父子进程会继承**同一套文件描述符表（FD table）**，这些 FD 指向内核里的“打开文件对象”（包含 pipe 端点对象）。
* 你代码里“子进程要关闭历史 wfd”就是直接利用/规避 fork 的这个特性。

1. **文件描述符（fd）是“进程视角的句柄”，真正对象在内核**

* `int fd` 只是表索引，真正的数据结构在内核：pipe buffer、引用计数等。
* 所以多个进程持有同一个管道写端，本质是对同一个内核 pipe 对象的**引用计数+1**。

1. **僵尸进程与 waitpid 回收**

* 子进程退出后会留下退出码等信息，直到父进程 `wait/waitpid` 回收才释放内核表项。
* 你 `CleanProcessPool()` 做的就是避免僵尸进程。

------

### B. IPC/匿名管道层（你代码“通信机制”的核心）

1. **匿名管道 pipe 的本质：内核中的单向字节流缓冲区**

* `pipefd[0]` 读端，`pipefd[1]` 写端
* 管道是一段内核缓冲区（队列），只提供**字节流**语义：写进去什么字节，读出来同样字节序列，不帮你分包。

1. **阻塞语义（默认阻塞）**

* `read`：缓冲区空会阻塞；除非写端全部关闭 -> `read` 返回 0（EOF）。
* `write`：缓冲区满会阻塞；如果读端都关闭，`write` 会触发 SIGPIPE 或返回 EPIPE。

1. **EOF 的判定规则（面试必问）**

> 对管道读端来说：只有当“所有写端引用计数归零”时，read 才会返回 0。
> 这句话就是你理解匿名管道 IPC 的“金句”。

你代码的退出机制就是：
父进程 `Close()` 所有写端 → 子进程 `read(0,...)` 得到 0 → break → exit → 父进程 waitpid。

1. **dup2 重定向：把 pipe 读端变成 stdin**

* 你在子进程里 `dup2(pipefd[0], 0)`，于是 Worker 里只需要 `read(0,...)`。
* 这相当于把“通信 fd”抽象成“标准输入”，面试官会觉得你思路清晰。

### C. 工程抽象层（你代码的“面向对象+解耦”）

## 2. 深挖底层：你的 IPC 在内核里到底发生了什么？

### 2.1 `pipe()` 创建了什么？

`pipe()` 创建的不是两个“文件”，而是一条内核 pipe 对象，里面包含：

* 一个**环形缓冲区**（实现上可能是 pipe buffer 链表/环）
* 两个端点：读端/写端
* 引用计数：有多少 fd 指向读端/写端

返回的 `pipefd[0], pipefd[1]` 只是把“读端点/写端点”登记到当前进程 fd 表里。

------

### 2.2 `fork()` 对 pipe 有什么影响？

fork 后：

* 父进程 fd 表复制一份给子进程
* 因此子进程也拥有这些 pipe 端点的 fd
* **内核 pipe 对象的写端引用计数变多**（关键）

这导致一个经典坑：

> 你以为父进程把写端关了子进程就会 EOF？
> 但实际上，如果**任意一个进程**还持有该 pipe 的写端（哪怕从来没写过），读端都不会 EOF。

所以你子进程里必须做：

```cpp
for(auto& c: _channels) c.Close();
```

这一步的意义是：
让子进程不再持有“历史所有管道写端”，否则 EOF 会被破坏（尤其是清理阶段）。

------

### 2.3 为什么你是一子进程一管道？（重要设计点）

你现在的结构是：**每个子进程一条专属管道**（父写、子读）。
它的好处（面试很加分）：

* **不需要锁**：不会出现多个子进程争抢同一条管道的读端，也不会出现多个写者竞争同一 pipe 的消息边界问题。
* **调度简单**：父进程只要选 `channels[i]` 写一个 int 就完事。

如果你用“多子进程共享一条管道”，会遇到：

* 多个 worker 同时 `read`，谁读到不可控（竞争）
* 需要定义更复杂的协议，甚至需要“消息队列”语义而不是字节流

你这个“一对一 pipe”是很合理的进程池入门架构。

------

### 2.4 `read`/`write` 的关键细节：你目前依赖了哪些“默认成立”的性质？

#### (1) “读到一个完整 int”在 demo 中几乎总成立

你写：

```cpp
int n = read(0, &cmd, sizeof(cmd));
if(n == sizeof(cmd)) ...
```

面试要知道：`read` **不是**保证读满，它可能返回 1~3 字节。
之所以你 demo 基本没出问题，是因为：

* 发送端一次写入 4 字节
* pipe 是本地内核缓冲
* 负载低，信号少

但规范上，正确写法应该循环读满 `sizeof(cmd)`（工程级）。

#### (2) “写一个 int 一次写完”在 pipe 通常成立

对 pipe 来说，写入长度 <= `PIPE_BUF` 时通常是原子写（不会和别的写交织）。
你这里更稳：**每条管道只有一个写者（父进程）**，所以不会交织。

#### (3) EINTR / SIGPIPE

* `read`/`write` 可能因为信号中断返回 -1 并置 `errno=EINTR`，需要重试
* 写到没有读端的 pipe，默认 SIGPIPE 可能直接把进程干掉
  你现在 demo 没遇到，但面试会问你是否知道这些坑。

## 3. 让你“深刻理解匿名管道”的三句铁律（背下来即可）

1. **管道是字节流，不是消息队列**：不会帮你分包，协议要自己定义。
2. **EOF 只在所有写端都关闭后才出现**：只要还有任何进程持有写端，读端永远不会返回 0。
3. **fork 会复制 fd**：所以必须在父/子进程各自关闭不需要的端，否则会泄露 fd、破坏 EOF、导致 waitpid 阻塞。

你代码已经把第 2、3 条用得很实，这是你最大的收获点。

## 4. 结合你的代码：为什么“清理阶段”最容易踩坑？

清理阶段的目标不是“wait 子进程”，而是两步：

### Step 1：让子进程有条件退出（触发 EOF）

* 子进程退出条件是 `read` 返回 0
* `read` 返回 0 的条件是：所有写端关闭
* 因此 master 必须关闭所有 `Channel` 写端

### Step 2：回收子进程（防僵尸）

* 退出后的子进程必须被 `waitpid` 回收

如果你**只做 Step 2，不做 Step 1**，就会出现：

* worker 还在 `read` 阻塞（因为写端没全关、或者被别的进程持有）
* 父进程 `waitpid` 永远等不到它退出 → 卡死

你注释里“version2 阻塞了”本质就是这个因果链：
**EOF 没触发成功 → 子进程不退出 → waitpid 阻塞。**

> 面试官追问时，你要把“引用计数导致 EOF 条件不满足”讲出来，这就是底层理解。
