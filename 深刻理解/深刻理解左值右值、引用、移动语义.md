# 左值和右值

* 左值(Lvalue)
  * 本质定义：有固定内存地址、有名字、**能活过当前这行代码**的对象，可以出现在赋值号`=`的两侧，是容器的本体，特征是能`&`取地址
* 右值（Rvalue）
  * 本质定义：没有固定内存地址、没有名字、**在这行代码结束后就会消亡**的临时数据，不能出现在赋值号`=`的左侧，是容器里的内容，特征是不能`&`取地址

举个例子：

```cpp
#include <iostream>
#include <string>

int main() {
    int a = 10;
    // a是左值，在栈上有确定的空间，有名字叫a
    // 10是右值，只是寄存器里的一个临时数字，用完即销毁
    std::cout << &a << std::endl;   // 左值可以取地址
    // &10; // 右值不能取地址

    std::string s1 = "hello";
    std::string s2 = "world";
    std::string s3 = s1 + s3;
    std::cout << &s1 << " : " << &s2 << " : " << &s3 << std::endl;
    // s1 s2 s3都是左值
    // (s1+s2)产生了一个没有名字的临时字符串，是右值，赋值给3后就销毁

    return 0;
}
```

还有个经典的例子：

```cpp
int x;
x++;
++x;
```

`x++` 和`++x`都是自增操作，但却是不同的值类别

**`x++`是右值：**

> 后置++操作中，编译器会先生成一份`x`的临时拷贝，然后才++，最后返回的是没有自加之前的那个临时副本

**`++x`是左值：**

> `++x`直接对`x`递增后立马返回`x`本身

再看个例子：

```cpp
int x = 1;
int getVal() { return x; }
void setVal(int val) { x = val; }
int main() {
    int y = getVal();
    steVal(2);
}
```

全局变量`x`原本是左值，但是经过`getVal`函数返回，`x`成了一个临时变量，变成了右值，**函数返回时，编译器不会返回对象本身，而是返回对象的临时拷贝**

在`main`函数中`setVal(2)`，实参2是一个右值，但是进入函数之后，形参`val`却变成了一个左值，**函数形参一定是左值**

# 左值引用和右值引用

* 左值引用`&`
  * 本质定义：给“活人”起个别名，必须绑定到左值上
* 右值引用`&&`
  * 本质定义：给”将死之物“（右值）**强行续命**，并打上标签：**我马上死了，我身上的资源可以随便抢**

怎么使用：

```cpp
// 左值引用和右值引用
// 坑：右值引用变量本身，是一个左值！
void func(int&& x) {
    // x是什么值？
    // 左值！
    // 虽然x的类型是右值引用，但是在函数内部有了名字x，也有了固定的内存地址
    std::cout << &x << std::endl;
}
int main() {
    int a = 10;
    int& ref1 = a;      // 正确：左值引用绑定到左值
    // int& ref2 = 20;  // 错误：普通的左值引用不能绑到右值上

    int&& r_ref1 = 20;  // 正确：右值引用绑定到右值，20被续命了，生命周期和r_ref1一样
    // int&& r_ref2 = a;// 错误：右值引用不能绑定到活着的左值上

    func(1);

    return 0;
}
```

# 移动语义

本质定义：

> C++11之前，我们要把一个临时对象（右值）赋给一个新对象时，智能去堆上开辟一块新内存，把数据**深拷贝**过去，然后销毁临时对象，代价太大了！
>
> **移动语义**，就是编译器发现这个对象是个**将死之物（右值）**，不再做深拷贝，而是直接**趁火打劫**，把右值底层的堆内存指针直接偷过来，然后把右值的指针置空

```cpp
// 移动语义
#include <cstring>
class MyString {
public:
    char* data;

    // 1. 传统的拷贝构造（极其昂贵）
    // 遇到左值时触发：我要完整克隆一份你的数据
    MyString(const MyString& other) {
        std::cout << "触发深拷贝！耗时操作！\n";
        data = new char[strlen(other.data) + 1];
        strcpy(data, other.data);
    }

    // 2. 现代的移动构造（零开销抢劫）
    // 遇到右值时自动触发：反正你马上要死了，你的堆内存直接归我了！
    MyString(MyString&& other) noexcept {
        std::cout << "触发移动语义！瞬间完成！\n";
        data = other.data;      // 1. 偷走底层堆内存指针
        other.data = nullptr;   // 2. 必须把原主人的指针置空，防止他析构时把内存炸了
    }

    ~MyString() { delete[] data; }
};
```

这里需要明确：

> **右值移动变量本身是左值没有错**->这是站在被调用方视角/表达式求值视角看待
>
> * **在函数体内部**，看大括号里的代码时，现在要关注的是**我该如何合法操作这个变量**，这个`other`变量已经分配了合法地址了
>
> **原本是右值也没有错**->这是站在调用方视角/空间初始化视角看待
>
> * **在函数体外部**，准备往函数里传参数的那一瞬间，此时关注的是**我传进去的是个什么状态的数据**

C++11提供了`std::move`强制转换类型的函数

注意：**`std::move`绝对不移动任何东西！底层一行汇编代码都不会产生！**

它的本质是：**把一个活着的左值，强制转换成一个右值引用，欺骗编译器触发移动语义**

当一个左值，**明确之后再也不用了**，就可以用`std::move`把它献祭出去

```cpp
MyString str1("Big Data");

// 我们想把 str1 的数据给 str2。
// 如果直接写 MyString str2 = str1; 会触发极其昂贵的深拷贝
// 但如果我确信 str1 以后不用了，我就可以主动交出所有权：
MyString str2 = std::move(str1); // 触发移动构造，瞬间完成！

// 坑点：此时 str1 已经被掏空了，它的 data 是 nullptr
// 再去访问 str1.data 会直接导致程序崩溃！
```

这里`std::move(str1)`整体是一个**将亡值(xvalue)**->那些**有物理实体（有内存），但马上就要被“抢劫”或销毁的对象**

我们用一个“器官捐献”的通俗例子来精准对应：

1. **纯右值（prvalue）**：无名无姓的克隆体。比如 `MyString("temp")`。它连身份证都没有，生下来就是为了被剥夺资源的
2. **左值（lvalue）**：比如函数里的 `other`。虽然它是走“右值通道”进来的，但只要它躺在医院的病床上，挂了号（有了变量名 `other`），有了病床号（有了内存地址），它在法律上就是一个活生生的**左值**。你不能直接抢它！
3. **将亡值（xvalue）**：怎样才能抢 `other` 的器官？你必须让它签署“放弃急救同意书”。**`std::move(other)` 就是在签这份同意书。** 签完之后，`std::move(other)` 这个**表达式的结果**，就是一个名副其实的**将亡值（xvalue）**，此时编译器才会无情地触发移动语义去掏空它

注意：**千万不要对一个`const`对象使用`std::move`**，`const`对象意味着不可以修改!

# 完美转发

## 举个案例理解痛点

* 业务场景：你要写一个代理函数`proxy`，它的任务是把外面传进来的参数，**原封不动**交给底层的`process`函数：外面传左值，就调用深拷贝；外面传右值，就调用移动语义

代码：

```cpp
void process(int& x)  { std::cout << "左值处理（深拷贝）\n"; }
void process(int&& x) { std::cout << "右值处理（零开销抢劫）\n"; }

// 作为一个极其平庸的中间商：
template <typename T>
void proxy(T&& arg) { 
    // 灾难发生：arg 进了门，有了名字，它变成了左值！
    // 无论外面传什么，这里永远只会调用 process(int& x)！
    process(arg); 
}

int main() {
    int a = 10;
    proxy(a);          // 期望输出：左值处理 -> 实际输出：左值处理（没毛病）
    proxy(20);         // 期望输出：右值处理 -> 实际输出：左值处理（血亏！性能暴跌！）
}
```

为啥！？

> 代理层`proxy`把参数的**原始值类别（左/右属性）给弄丢了！**

## 万能引用和引用折叠

要解决这个问题，就需要让`proxy`代理函数**同时接收**左值和右值，并且让底层明确

在模板函数`template <typename T> void proxy(T&& arg)`中，`T&&`绝对不是右值引用！，它是**万能引用**，传左值它就是左值引用，传右值它就是右值引用

引用折叠规则：

> 编译器在推导类型 `T` 时，会记录下原始参数的身份。由于 C++ 不允许出现“引用的引用”（比如 `int& &&`），所以出台了一套**折叠规则（左值引用是强势基因，只要有 `&`，结果就是 `&`）**
>
> * `&` + `&` = `&` （传左值）
>
> * `&` + `&&` = `&` （传左值）
>
> * `&&` + `&` = `&` （传左值）
>
> * **`&&` + `&&` = `&&` （只有纯右值相遇，才保留右值属性）**

当执行 `proxy(a)` 时，`T` 被推导为 `int&`，折叠后 `arg` 就是 `int&`

当执行 `proxy(20)` 时，`T` 被推导为 `int`，折叠后 `arg` 保持 `int&&`

## `std::forward`完美转发

**本质定义**：`std::forward<T>()` 是一个**条件类型转换（Conditional Cast）**

* 它的逻辑极其粗暴：它去翻看模板类型 `T` 的底单
* 如果 `T` 显示它原来是个左值，`forward` 就什么也不做，让它继续当左值
* 如果 `T` 显示它原来是个右值，`forward` 就**顺手给它套一个 `std::move()`**，把它强转回将亡值（右值）！

```cpp
void process(int& x)  { std::cout << "左值处理（深拷贝）\n"; }
void process(int&& x) { std::cout << "右值处理（零开销抢劫）\n"; }
template <typename T>
void proxy(T&& arg) { // T&& 是万能引用，吸纳一切
    // 使用 std::forward<T> 恢复其出厂设置，完美转交给下一层
    process(std::forward<T>(arg)); 
}

int main() {
    int a = 10;
    proxy(a);  // 输出：左值处理（完美）
    proxy(20); // 输出：右值处理（完美！成功触发移动语义！）
}
```

# 总结

![img](https://gitee.com/binary-whispers/pic/raw/master///20260305212938824)

* **横轴（有名字 vs 没名字）**：在底层，它代表这个数据**有没有身份**。也就是说，它在内存里有没有一个稳定的“门牌号”（能否用 `&` 取地址），程序能不能在下一行代码继续找到它
* **纵轴（不能移动 vs 能移动）**：在底层，它代表这个数据**有没有被抢劫的许可**。也就是说，编译器是否允许调用移动构造/移动赋值，把它底层的堆内存指针直接偷走
