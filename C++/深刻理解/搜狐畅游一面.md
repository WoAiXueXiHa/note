## 1. 智能指针是什么？有哪几类？实现思路？

可直接回答：

智能指针本质上是一个类，它内部持有一个原始指针，并且通过 RAII 的方式，在离开作用域时自动释放资源，同时重载了 `*` 和 `->` 操作符，让使用方式接近普通指针。这样可以减少手动 `new/delete` 带来的内存泄漏、重复释放等问题。

标准库里主要有三类：

* `std::unique_ptr`：独占所有权，禁止拷贝，只能移动，适合一个对象只被一个地方管理的场景。
* `std::shared_ptr`：共享所有权，通过引用计数管理同一块堆内存，最后一个 `shared_ptr` 析构时释放对象。
* `std::weak_ptr`：弱引用，不增加引用计数，常配合 `shared_ptr` 用来打破循环引用，比如观察者模式或双向关系。

实现思路上，以 `shared_ptr` 为例，一般会有两部分：

1. 一个指向实际对象的指针 `T* ptr`；
2. 一个指向控制块的指针，控制块里保存引用计数、弱引用计数，以及删除函数。

拷贝一个 `shared_ptr` 时，就是让两个 `shared_ptr` 都指向同一个控制块，并把控制块里的引用计数做一次原子加一；析构时做原子减一，如果减到 0，则释放对象，再看弱引用计数是否也为 0，如果是就连同控制块一起释放。`unique_ptr` 则只需要在析构时调用 `delete`，并通过删除拷贝构造和拷贝赋值来保证独占所有权。

## 2. `shared_ptr` 是线程安全的吗？怎么实现线程安全？

可直接回答：

`shared_ptr` 的线程安全性可以分两部分说：

第一，对同一块对象的生命周期管理是线程安全的。也就是说，不同线程各自持有一个 `shared_ptr` 的拷贝，哪怕同时构造、拷贝或者析构，它内部的引用计数操作是通过原子操作或内部锁来保护的，不会出现引用计数错乱、对象被提前释放或者永远不释放的情况。

第二，但对同一个 `shared_ptr` 变量本身的读写不是线程安全的。如果多个线程同时读写同一个 `shared_ptr` 变量，比如一个线程在重置它，另一个线程在拷贝它，这个操作需要外层再加锁，否则会有数据竞争。

底层实现上，`shared_ptr` 通过一个控制块来管理引用计数，控制块里的 `use_count` 通常是 `std::atomic<long>`。每次拷贝构造或者拷贝赋值的时候，会对 `use_count` 做一次原子 `fetch_add`；每次析构做一次原子 `fetch_sub`，如果减到 0，就调用删除器释放所管理的对象。这个原子计数是线程安全的，所以对象的销毁时机在多线程环境下是可靠的。

## 3. 多态的实现条件？

可直接回答：

如果说的是运行时多态，一般需要满足三个条件：

1. 基类中至少有一个虚函数；
2. 派生类对这个虚函数进行重写（`override`），保持函数签名一致；
3. 通过基类的指针或者引用指向派生类对象，并通过这个指针或引用调用虚函数。

在这样的条件下，编译器就会通过虚函数表和虚函数指针来实现动态绑定。调用虚函数时，会根据对象实际的动态类型，去它的虚表里找到对应版本的函数，从而在运行时决定真正调用的是基类实现还是派生类实现，这就是运行时多态。

## 4. 什么是虚函数？实现原理？

可直接回答：

虚函数就是在基类中用 `virtual` 关键字修饰的成员函数，用来实现运行时的动态绑定。当我们通过基类指针或引用调用虚函数时，实际调用的函数版本会根据对象的动态类型来决定，而不是根据指针的静态类型。

典型的实现原理是虚函数表机制。对于包含虚函数的类，编译器会为它生成一张虚函数表，表里按固定顺序存放着这个类及其基类的虚函数地址。每个对象内部，会隐式多出一个虚表指针 `vptr`，指向所属类的虚函数表。当我们写 `Base* p = new Derived; p->func();` 时，实际执行过程大致是：通过 `p` 访问对象里的 `vptr`，再从虚表中找到 `func` 对应的槽位，取出函数指针并调用。如果 `Derived` 重写了这个虚函数，那么表里存的就是 `Derived::func` 的地址，从而实现了多态。

## 5. 父类析构函数必须是虚函数吗？为什么？

可直接回答：

如果一个类会被当作多态基类使用，也就是我们会通过 `Base*` 指向 `Derived` 对象，并通过 `delete Base*` 去释放，那么基类的析构函数必须声明为虚函数。

原因是，只有把析构函数声明成虚函数，它才会出现在虚函数表里。这样当我们通过 `Base*` 调用 `delete` 时，实际上会通过 `vptr` 在虚表里找到真正的析构函数入口，先调用派生类的析构，再调用基类析构，整个析构链是完整的。反过来，如果基类析构不是虚函数，那么 `delete Base*` 时，只会调用基类的析构函数，派生类那一部分资源得不到释放，会造成内存泄漏甚至未定义行为。

如果一个类完全不打算作为多态基类使用，比如只是一个简单的值类型或者工具类，那可以不把析构设成虚函数，节省一点虚表相关的开销。但一旦涉及“通过基类指针删除派生类对象”，就应该把基类析构声明为虚函数。

## 6. `static` 的使用（变量、函数、类成员）

可直接回答：

`static` 在 C++ 里主要有三个层面的含义：存储期、链接性和作用域，是跟“生命周期”和“可见性”相关的关键字。

* 对于函数内部的 `static` 局部变量，它只会在第一次执行到定义处时初始化一次，生命周期从那一刻开始直到程序结束，但作用域依然只在函数内部。常用在做缓存、单例之类的场景。
* 对于类的 `static` 成员变量，它属于类本身，不属于某个具体对象实例，所有对象共享同一份数据。通常需要在类外提供一个定义（或者使用 `inline static`）。这种成员可通过 `ClassName::member` 直接访问。
* 在全局或命名空间范围上给变量或函数加 `static`，改变的是它的链接属性，从外部链接变为内部链接。也就是说，这个符号只在当前编译单元内部可见，别的源文件无法通过 `extern` 引用到它，相当于文件内“私有”。
* 对于类的 `static` 成员函数，没有隐含的 `this` 指针，只能访问静态成员或通过参数访问对象数据，常用来实现与类相关但不依赖于特定对象状态的功能。

## 7. `vector` 的扩容机制？怎么避免频繁扩容？

可直接回答：

`std::vector` 底层是连续内存的动态数组，它一般维护三根指针：起始位置、已用结束位置和容量结束位置。当插入新元素时，如果当前 `size < capacity`，就直接在尾部构造元素；如果 `size == capacity`，就需要扩容。

扩容时的典型流程是：

1. 按某个增长策略（通常是 2 倍增长，标准没硬性规定）计算出一个更大的容量；
2. 向堆上申请一块新的连续内存；
3. 把原有元素用拷贝或移动构造迁移到新内存；
4. 调用原内存中元素的析构函数，并释放旧内存；
5. 更新内部指针。

这一步的时间复杂度是 O(n)，而且涉及大块内存搬迁，所以如果在插入过程中频繁触发扩容，会有明显的性能损耗。避免频繁扩容的最好方式是，在知道大致元素数量时，提前调用 `reserve` 预留足够容量，比如读文件或做网络缓存的时候可以 `v.reserve(expected_count)`，这样在插入过程中就很少甚至不会再触发扩容。

## 8. `map` 和 `unordered_map` 的底层数据结构

可直接回答：

`std::map` 底层通常实现为一棵红黑树，也就是自平衡二叉搜索树。这样可以保证元素按 key 有序存储，查找、插入和删除的时间复杂度在平均和最坏情况下都是 O(log n)，同时支持按序遍历。

`std::unordered_map` 底层是哈希表，标准没规定具体实现，一般是“数组 + 链表”或者“数组 + 开放定址”。它根据 key 的哈希值把元素分配到不同的桶里。在哈希函数较好、装载因子控制得当的情况下，查找、插入、删除的平均时间复杂度是 O(1)，但最坏情况下可能退化到 O(n)。因为按桶存放的，所以遍历顺序不保证和 key 的大小有关。

## 9. 怎么避免 / 处理哈希冲突？

可直接回答：

哈希冲突其实是不可完全避免的，只能从“减少冲突”和“处理冲突”两个层面来回答。

减少冲突的角度，一方面要设计一个足够好的哈希函数，让不同 key 尽量均匀地映射到各个桶；另一方面要控制装载因子，也就是元素数量和桶数量的比例，当装载因子过大时通过 rehash 扩充桶的数量，降低每个桶里的平均元素数。

在处理冲突的方式上，常见的有两大类：

* 一类是开放定址法，当某个位置发生冲突时，就根据一定的探查策略继续在表中找空位，比如线性探查、一致性探查或者双重哈希。查找时则沿着同样的探查序列向后查，直到找到目标或者遇到空位。
* 另一类是链地址法，也就是每个桶对应一条链表或者更高级的数据结构（比如红黑树），发生冲突的元素就插入到这条链表中。`std::unordered_map` 一般就是数组 + 链表的结构，通过 rehash 来控制平均桶长度。

这两种方案都在实际工程里被广泛使用，各有空间和性能上的 trade-off。

## 10. 有哪些排序算法？时间复杂度分别是多少？

可直接回答：

常见的排序算法里，我一般会按照复杂度来记：

* 冒泡排序、选择排序、简单插入排序，这三种的平均和最坏时间复杂度都是 O(n²)，只适合小规模数据或者几乎有序的场景。其中插入排序在序列基本有序时最好可以到 O(n)。
* 归并排序的时间复杂度在平均和最坏情况下都是 O(n log n)，而且是稳定排序，但需要 O(n) 的额外空间。
* 快速排序在随机数据上的平均时间复杂度是 O(n log n)，但如果选枢轴不当会退化到 O(n²)。实际实现中通过随机化枢轴、三路划分等手段，可以大大降低退化概率。快速排序是不稳定的。
* 堆排序的平均和最坏时间复杂度也都是 O(n log n)，不稳定，但只需要 O(1) 的额外空间。它的核心是用一个大根堆来不断取出当前最大值。

在标准库里，`std::sort` 通常会使用改进版的快速排序或 introsort（快排 + 堆排 + 插排），综合考虑了性能和最坏情况。

## 11. 堆排序的具体思路？

可直接回答：

堆排序的核心思想是先把数组看成一个完全二叉树，然后调整成一个大根堆，堆顶永远是当前最大值。整体可以分为两步：

第一步是建堆。把数组下标当作完全二叉树的节点，父节点是 `i`，左孩子是 `2*i+1`，右孩子是 `2*i+2`。从最后一个非叶子结点开始（`n/2 - 1`），一路向前，对每个结点做一次 `heapify` 下滤操作，让以它为根的子树满足大根堆性质。建堆的总复杂度是 O(n)。

第二步是排序。此时 `A[0]` 是当前最大元素。我们把 `A[0]` 和 `A[heap_size-1]` 交换，把堆的大小减一，最大元素就被“固定”在数组末尾。然后对新的堆顶再次 `heapify`，恢复大根堆性质。重复这个过程，直到堆大小减到 1，整个数组就是按从小到大的顺序排好。

`heapify` 本身就是不断比较当前结点和左右孩子，选出最大者，如果孩子更大就交换下去并递归或循环处理它的孩子，时间复杂度是 O(log n)。

## 12. 进程和线程的区别？

可直接回答：

进程可以看作是操作系统分配资源的基本单位，每个进程有自己独立的虚拟地址空间，包括代码段、数据段、堆和各自的栈，还有自己的文件描述符表和其他内核资源。不同进程之间默认是相互隔离的，需要通过进程间通信机制来交换数据。

线程是 CPU 调度的基本单位，一个进程内可以有多个线程，它们共享这个进程的地址空间和大部分资源，比如代码段、全局变量、堆和打开的文件描述符，但每个线程有自己独立的栈和寄存器上下文。这样线程之间切换比进程切换更轻量，但由于共享内存，也更加容易因为缺少同步而产生数据竞争或者把整个进程搞崩。

总结一句：进程是资源隔离的边界，线程是执行流的单位，同一进程内的线程通信更高效但更容易出错，不同进程之间通信更安全但需要依赖 IPC 机制。

## 13. 线程同步怎么实现？

可直接回答：

线程同步的核心目的是在多个线程访问共享数据时，避免数据竞争和竞态条件。常见的实现方式主要有几类同步原语：

第一是互斥锁，比如 `std::mutex`。它保证同一时刻只有一个线程可以进入临界区，典型用法是 `lock_guard` 或 `unique_lock`。底层一般会用原子指令配合内核的 futex，在竞争不严重时自旋，在竞争严重时把线程挂起。

第二是读写锁，比如 `std::shared_mutex` 或 `pthread_rwlock`。读锁可以多个线程同时持有，写锁是独占的，适合读多写少的场景，减少写锁带来的阻塞。

第三是条件变量，比如 `std::condition_variable`。搭配互斥锁使用，线程在条件不满足时调用 `wait` 阻塞，被放入等待队列，当条件满足时由其他线程调用 `notify_one/notify_all` 唤醒，用来实现生产者-消费者等模式。

第四是原子操作，比如 `std::atomic`。可以在不显式加锁的情况下，通过比较交换（CAS）等原子指令实现无锁的计数器或者无锁队列，这样在高并发场景下能减少锁竞争，但实现会更复杂。

具体选择哪种同步方式，要根据共享数据访问模式（读多写少、延迟敏感、是否追求无锁）来权衡。

## 14. `socket` 创建之后怎么设置端口复用？

可直接回答：

在 Linux 下，我们通常在 `bind` 之前，通过 `setsockopt` 设置端口复用相关的选项。典型代码大概是这样：

int fd = socket(AF_INET, SOCK_STREAM, 0);

int opt = 1;

*// 地址复用：允许处于 TIME_WAIT 状态的地址重新绑定*

setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

*// （可选）端口复用：允许多个 socket 绑定到同一 ip:port，用于多进程/多线程负载均衡*

setsockopt(fd, SOL_SOCKET, SO_REUSEPORT, &opt, sizeof(opt));

bind(fd, (sockaddr*)&addr, sizeof(addr));

listen(fd, SOMAXCONN);

`SO_REUSEADDR` 主要解决的是服务器重启时端口被 TIME_WAIT 占用的问题，可以更快地重新绑定同一端口；`SO_REUSEPORT` 是 Linux 3.9 以后提供的，它允许多个 `socket` 绑定到同一 `ip:port`，常用在多进程或多线程模型，每个线程一个监听 `socket`，由内核做负载分配，能减少锁竞争。

## 15. 连接建立后，把网线拔掉，上层会有感知吗？

可直接回答：

默认情况下，TCP 不会立刻把“网线被拔掉”这类物理层故障反馈给应用层。连接建立好之后，如果对端突然断电或者物理链路断掉，本地内核通常感知不到，只要本端不去收发数据，这个 TCP 连接在内核视角还是“已建立”的。

应用层一般会在两种情况下感知到问题：

一是继续发送数据。此时内核会尝试重传，如果在重传次数和时间阈值内一直得不到确认，最后会认为连接超时或不可达，`send` 或 `write` 返回错误，比如 `ETIMEDOUT` 或 `ECONNRESET`。

二是启用了 TCP keepalive 或者应用层自己做心跳。开启 `SO_KEEPALIVE` 并设置合理的探测间隔后，内核会定期发送探测包，如果多次探测失败，也会关闭连接，这时上层在下一次操作时会收到错误。很多高可用系统会在应用层自己定义心跳协议，比如固定间隔发送 ping 包，如果超过几次没收到回应就主动关闭连接。

所以更准确地说：拔掉网线不会被立即感知，只有在我们去收发数据或者依赖 keepalive/心跳探测时，才会在一定延迟后通过错误码间接察觉到连接已经不可用。
